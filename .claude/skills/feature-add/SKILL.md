# feature-add Skill

## 触发条件

当用户提到新增特性，新增功能，新增需求或类似表达时（就是在开发中会被认为是feature的修改开发）

> 触发时必须确保已经阅读了"\.claude\rules\开发指南.md"，并遵循其中的规范，本文是修改时的具体过程细节

## Feature Add流程
1. 按照plan-management.md的规则新增一个M计划，然后按实际情况制定修改计划，其中必须包括的S级别计划有：阅读并修改PRD中的对应内容，查找到对应的Architecture类和TDD类文档，修改Achitecture类和TDD类文档的对应内容，修改代码，验收收尾。中间可以按需插入所需的S级Plan或者T级别的Plan。
2. 找到PRD中有关新feature的全部相关地方(按照"\.claude\skills\docs-generator\references\prd-template.md"的要求在PRD文档中符合规范的合理的放入这个feature的相关内容，比如其对应的需求，以及所属的概念或机制，或者纯新的概念和机制，总之就是按照prd-template的那些进行拆解新feature并融入在原有的PRD中)，找到后暂停告诉用户PRD中关于feature需要修改新增的所有点，以及解决方案，等待用户确认
3. 用户确认后，你找到PRD对应的Architecture文档（通过调用codebase-index知道如何找到对应Architecture部分），找到文档中新feature可能有关的部分（就是从技术层面拆解feature的具体实现，并且和现有已经实现的进行比对，如果有类似的原子能力已经被实现或者有相同功能或者类似功能，那就可以复用（如果需要也可在争求用户同意后修改原有的部分，使其可以被复用，或者融入新的feature），然后修改最初的feature实现方案，如此迭代），找到后暂停告诉用户你找到bug对应的Architecture文档的脉络路径以及Architecture类文档中关于feature需要修改的所有点(为什么要这么修改，怎么符合原有架构设计思想的)，以及Architecture文档中的相关部分打算如何修改，等待用户确认
4. 用户确认后，你找到Architecture对应的TDD文档（通过调用codebase-index知道如何找到对应TDD部分），找到文档中关于feature的部分，找到后暂停告诉用户你找到feature对应的TDD类文档的脉络路径以及Architecture类文档中关于feature需要修改的所有点（设计细节），以及TTD类文档中的相关部分打算如何修改，等待用户确认
5. 找到所有所需的Architecture和TDD文档类并修改后，按照Architecture和TDD的对应要求开始修改代码
6. 在修改和实测过程中可能会有大大小小的和对应的Architecture类和TDD类的对应内容有不一样的地方，没有关系，按照你实际的修改和实测结果修改进行，并在验收收尾的S级plan中将对应的和原来Architecture类和TDD类文档不一样的地方给修改过来了，并告知用户

## 规则
融入新feature的时候一定要注意不要影响到了其它与之有耦合的功能，你可以调用lsp和serena帮助你进行确认
融入feature要遵循代码本身的架构设计和规范，以及考虑复用性，对于相同的功能的代码，可以考虑修改相同功能的代码，使其可以被复用
