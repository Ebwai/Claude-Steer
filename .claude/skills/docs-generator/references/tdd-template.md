# TDD 模板 (Technical Design Document)

## 核心定位

TDD 回答「**每个模块如何实现？（Implementation）**」，聚焦**技术实现**，不讨论产品价值。

---

## 章节结构

对 Architecture 中定义的每个模块，逐模块展开以下章节：

### 1. Module Overview（模块概览）

- **职责**：这个模块的职责是什么
- **输入**：模块接收什么数据/信号（从哪些来源，什么格式）
- **输出**：模块产出什么数据/信号（输出到哪里，什么格式）

### 2. API（接口定义）

- **接口签名**：完整的函数签名/方法签名/API endpoint
- **参数**：每个参数的类型、含义、是否必填、默认值
- **返回值**：返回值的类型和含义
- **错误码/异常**：可能返回的错误类型及含义
- **调用示例**：典型调用方式

### 3. Data Model（数据模型）

可以包含可选的四部分(按需配置，没有就不用写)：

#### 3.1 Table（有哪些表/数据结构）

```markdown
| 表/结构名 | 说明 | 存储引擎/类型 |
|-----------|------|--------------|
| [名称] | [说明] | [如 MySQL, SQLite, 内存, 文件等] |
```

#### 3.2 Field（每个字段是什么）

对每个表/结构：

```markdown
### [表名]

| 字段名 | 类型 | 必填 | 默认值 | 说明 | 约束 |
|--------|------|------|--------|------|------|
| [字段1] | [类型] | 是/否 | [默认值] | [说明] | [约束条件] |
```

#### 3.3 Index（哪些字段需要索引）

```markdown
| 索引名 | 字段 | 类型(唯一/普通/全文等) | 说明 |
|--------|------|----------------------|------|
| [索引名] | [字段列表] | [类型] | [为什么需要这个索引] |
```

#### 3.4 Migration（数据库如何升级）

```markdown
### Migration: [版本号]

- **变更类型**：新增表 / 修改字段 / 新增索引 / 数据迁移
- **变更内容**：[具体变更描述]
- **回滚方案**：[如何回滚此变更]
- **影响范围**：[影响哪些模块和功能]
```

### 4. Sequence（时序/流程）

用文字或时序图描述关键操作的执行时序，这部分每个操作的时序单独写一个xxxSOP.md，然后在本文中每个关键操作的xxxSOP.md：

- **正常流程**：一次完整调用的步骤顺序
- **异常流程**：出错时的回退/补偿步骤
- **并发考虑**：多线程/多请求同时执行时的行为

### 5. State Machine（状态机）

对有状态的实体，定义：

- **状态列表**：该实体可能处于的所有状态
- **状态转换条件**：从状态 A 到状态 B 需要满足什么条件
- **状态转换副作用**：状态切换时触发什么操作

示例格式：

```markdown
### 状态机: [实体名称]

**状态**：
- `状态1` — [说明]
- `状态2` — [说明]

**转换规则**：
| 从 | 到 | 触发条件 | 副作用 |
|----|-----|---------|--------|
| 状态1 | 状态2 | [条件] | [副作用] |
```

### 6. Exception（异常处理）

- **可恢复异常**：什么异常可以自动恢复，恢复策略是什么
- **不可恢复异常**：什么异常需要人工介入，如何通知
- **降级策略**：关键路径上某个模块不可用时的降级行为
- **重试策略**：什么情况需要重试，重试次数、间隔、退避算法

### 7. Monitoring（监控）

**所有必须添加日志的点，以及如何利用日志来 trace：**

```markdown
### 关键日志点

| 日志点 | 级别 | 记录内容 | 用途 |
|--------|------|---------|------|
| [位置/事件] | DEBUG/INFO/WARN/ERROR | [记录什么信息] | [排查什么问题] |

### Trace 方案

- **Trace ID 传递**：[如何生成、如何在模块间传递 trace id]
- **关键指标**：[需要监控的关键指标及告警阈值]
```

### 8. Testing（测试）

需要验证的关键节点：

```markdown
### 测试清单

| 测试点 | 测试类型（单元/集成/E2E） | 测试场景 | 预期结果 |
|--------|--------------------------|---------|----------|
| [测试点1] | [类型] | [场景描述] | [预期结果] |

### 关键验证节点

- [列出最关键的、必须通过测试验证的功能节点]
```

---

## 约束

- **不讨论产品价值**：不解释「为什么用户需要这个功能」（已在 MRD/PRD 中讨论）
- 聚焦具体的技术实现细节
- 所有 API、数据模型、状态机必须与 Architecture 中定义的模块边界一致
- 不确定的技术细节标注 `[待确认]`
