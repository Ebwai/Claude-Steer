# Architecture Design 模板

## 核心定位

Architecture Design 回答「**系统应该如何组织？（System）**」，聚焦**系统架构**，不讨论接口细节和数据库细节（留到 TDD）。

---

## 章节结构

### 1. Overall Architecture（整体架构）

- **架构风格**：采用什么架构风格/模式（如分层架构、微服务、事件驱动、插件化等）
- **架构图**：用文字或 mermaid 描述系统的顶层结构
- **设计理念**：为什么选择这种架构风格？根据PRD的什么特点，这个架构可以解决什么问题？
- **关键设计决策**：最重要的 3-5 个架构决策及其理由

### 2. Module Boundary（模块边界）

**只定义职责，不写 API。**

对每个模块描述：

- **模块名称**
- **职责**：这个模块负责什么（一句话）
- **边界**：这个模块不负责什么，哪些事情明确不属于它
- **依赖**：这个模块依赖哪些其他模块（仅列出模块名，不列具体接口）

示例格式：

```markdown
### 模块: [模块名]

- **职责**：[一句话描述该模块的核心职责]
- **边界**：
  - 负责：[列出该模块负责的事情]
  - 不负责：[列出明确不属于该模块的事情]
- **依赖**：[依赖的其他模块名称列表]
```

### 3. Component Interaction（组件交互）

- **组件间通信方式**：各模块/组件之间如何通信（如同步调用、异步消息、事件总线、共享状态等）
- **交互模式**：主要的交互模式（如请求-响应、发布-订阅、管道-过滤器等）
- **关键交互场景**：选几个最重要的交互场景描述

### 4. Data Flow（数据流）

- **数据流向**：数据从入口到出口的完整流动路径
- **数据转换点**：数据在哪些节点发生格式/结构转换
- **数据存储策略**：哪些数据需要持久化、哪些是瞬时的（只说要存什么类型的数据，不涉及表结构）
- **常见的使用场景的完整数据流**

### 5. Technology Selection（技术选型）

对每个关键技术选择：

- **技术**：选用的技术/框架/库
- **用途**：在项目中承担什么角色
- **选型理由**：为什么选它而非替代方案（考虑因素：成熟度、社区、性能、团队能力、跨平台支持等）
- **替代方案**：考虑过但未选的方案及原因

### 6. Non-functional Design（非功能性设计）

- **高可用（Availability）**：如何保证系统可用的设计
- **解耦性**
- **复用性**
- **可扩展（Scalability）**：系统如何应对增长的设计
- **可观测性（Observability）**：日志、监控、追踪的整体策略
- **跨平台（Cross-platform）**：如何在多平台间保持一致性和适配差异（如适用）

---

## 约束

- **不讨论 API 细节**：不定义具体的 API 签名、参数列表、返回值（留到 TDD）
- **不讨论数据库细节**：不定义具体的表结构、字段类型、索引（留到 TDD）
- **不讨论产品价值**：不解释「为什么用户需要这个功能」（已在 MRD/PRD 中讨论）
- 聚焦系统层面的组织结构和模块关系
