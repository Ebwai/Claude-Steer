# T1-T2 — 定时触发实现（合并）

## 技术原理

定时触发利用 Claude Code 的 `/loop` 斜杠命令实现跨重启持久化的定时任务。

### /loop 工作机制

- `/loop {interval} {prompt}` — 在当前会话内创建周期性任务（使用 CronCreate 工具）
- 任务文件存储在 `~/.claude/tasks/<session-uuid>/<task-id>.json`
- 会话结束后任务**仍持久**，通过 `claude --resume <claudeId>` 恢复会话后任务继续执行
- 任务 **7 天后自动过期**；每会话最多 50 个任务
- interval 格式：`5m`、`1h`、`1d` 等

### 架构设计

```
一个项目 → 一个 loop session (PTY)
同一项目的多个定时任务 → 同一 PTY 内连续发送多条 /loop 命令
```

**本地持久化**（`~/.claude-driver/scheduler-sessions.json`）：
```json
{
  "projectPath": "/home/tony/myproject",
  "claudeId": "abc123...",
  "tasks": [
    { "taskId": "uuid", "interval": "1h", "prompt": "检查代码", "createdAt": 1234567890 }
  ]
}
```

每个 session entry 以 `claudeId` 为唯一键（projectPath 可能变化，但同一项目一定同一 claudeId）。

### UI 结构

点击「⏰ 定时触发」→ 弹出 Modal（与 SoulModal 同层），包含两个 Tab：
- **Claude介入** — 本次实现
- **脚本触发** — 显示「开发ing，敬请期待」

### 任务生命周期

1. 用户填入：项目（下拉选认领项目）+ 间隔 + 提示词 → 点击「创建」
2. 主进程检测该项目是否已有活跃 loop session（根据持久化 claudeId 检查）：
   - **有活跃 PTY** → 直接向该 PTY 发送 `/loop {interval} {prompt}\r`
   - **无活跃 PTY** → 启动 `startBare(...)` 以裸 `claude` 命令在项目目录开启 PTY，等待提示符后发 `/loop`
   - **有 claudeId 但 PTY 未运行** → `startBare` 以 `claude --resume <claudeId>` 启动，等待提示符后发 `/loop`
3. 监听 PTY stdout，从中解析 `PTY_BIND`（或检测 sessionId 绑定），待 `/loop` 发出成功后（stdout 含 loop 确认字样），推送任务到 renderer
4. **任务在方案B时显示**：PTY 准备好且 /loop 成功发送后，任务才出现在列表

### Toggle 语义

- **ON**（绿色）= PTY 运行中，loop 任务活跃
- **OFF**（灰色）= 停止该 PTY（`stopSession`）；任务记录保留在本地持久化，显示「已暂停」
- 切换为 ON = 重新 `resume` 该 claudeId 的会话

### 过期任务

- 通过 `createdAt + 7天` 计算是否过期
- 过期任务：隐藏 Toggle，显示「重新创建」按钮（重新发 `/loop`，在同一或新建会话中）

## 实现步骤

### 原子目标 T1 — IPC 通道 + 主进程 handler [x]

**新增 IPC 通道**（`ipc-channels.ts`）：
```typescript
SCHEDULER_LIST:   'scheduler:list',    // 读取本地持久化 + PTY 活跃状态
SCHEDULER_CREATE: 'scheduler:create',  // 创建新 loop session 或向已有 session 追加任务
SCHEDULER_TOGGLE: 'scheduler:toggle',  // 暂停/恢复单个项目的 loop PTY
SCHEDULER_DELETE: 'scheduler:delete',  // 删除一个任务记录（仅本地，不停止 PTY）
```

**主进程逻辑**（`main/index.ts`）：
- 模块级：`const schedulerPtyIds = new Set<string>()` — 与 insightPtyIds 同类机制隔离 hook 路由
- 模块级：`const schedulerSessionMap = new Map<string, string>()` — projectPath → ptyId（运行时状态）
- `SCHEDULER_LIST` handler：读 `~/.claude-driver/scheduler-sessions.json`，返回带活跃状态的任务列表
- `SCHEDULER_CREATE` handler：
  1. 查是否有该项目的活跃 PTY
  2. 无则 startBare（裸 claude 或 --resume）
  3. 等待提示符（❯/> /?) → 发 `/loop {interval} {prompt}\r`
  4. 等待 stdout 确认（含 "loop" 或 "Created" 等）
  5. 写入持久化文件，返回 `{ ok: true, taskId }`
- `SCHEDULER_TOGGLE` handler：停止/恢复对应项目 PTY
- `SCHEDULER_DELETE` handler：从持久化文件中删除指定 taskId

## 数据类型

```typescript
interface SchedulerTask {
  taskId: string
  projectPath: string
  projectName: string     // 显示用，从 projects atom 查
  claudeId: string        // 该项目 loop session 的 claude session id
  interval: string        // "1h"
  prompt: string
  createdAt: number       // unix ms
  isActive: boolean       // 对应 PTY 是否运行中
  isExpired: boolean      // createdAt + 7天 < now
}
```

## 验收标准

- 选择项目 + 填间隔 + 填提示词 → 点「创建」→ 任务列表出现（仅 PTY 就绪且 /loop 发出后显示）
- Toggle 关闭 → PTY 停止，列表显示「已暂停」
- Toggle 打开 → PTY 恢复，列表显示「运行中」
- 同一项目可以有多个任务条目，但底层共用一个 PTY
- 7 天过期的任务：隐藏 Toggle，显示「重新创建」按钮
- 脚本触发 Tab 显示「开发ing」占位

## 废弃计划（原旧版本）

**废弃原因**：原版依赖三种方式对比卡片（Cloud/Desktop/Session），与用户实际需求不符。
用户需求聚焦于「Claude介入 + 脚本触发」两个维度，且应以 Modal 形式而非内嵌面板呈现。
