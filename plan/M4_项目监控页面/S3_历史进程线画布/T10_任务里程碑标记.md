# T10 — 任务里程碑标记（Milestone）

> **更新日期**：2026-04-25  
> **更新原因**：原计划基于旧架构设计（main 进程 PlanSyncService、独立 IPC 通道、三列布局），与实际已实现代码结构不符。本次完整重写。

---

## 技术原理

每个 Session 框左侧绝对定位显示任务里程碑标记（Milestone Badge），
当某一个 **T 级 Plan 节点**状态由非 DONE 变为 DONE 时，在当前进程线快照位置插入对应的紫色 badge。

### Milestone Badge 规格

- 格式：`[M4-S3-T10]`（对应 PlanNode.id）紫色背景 badge，字体 9px
- 位置：绝对定位在 `.sfn-timeline-area` 内，`left: 4px`（紧贴框左边），`top` 由 `nodeIndex` 估算
- hover：显示该任务的完整描述文字（PlanNode.title）
- 随框内容一起绘制（非 fixed 定位），不随 ProcessTimeline 滚动而消失（ProcessTimeline 在 SessionFrameNode 内本身不滚动，框本身随内容增高）

### Y 坐标估算公式（与 BranchEdge 相同模式）

```
top = FRAME_HEADER_HEIGHT + TIMELINE_PADDING_TOP + nodeIndex × NODE_HEIGHT_ESTIMATE
```

- `FRAME_HEADER_HEIGHT = 40`（来自 useSessionFrameLayout.ts 实际值，框头部高度）  
- `TIMELINE_PADDING_TOP = 16`（来自 ProcessTimeline.css，`.ptl-container` 的 padding-top）  
- `nodeIndex`：Milestone 创建时刻，`timelineBySessionAtom(sessionId)` 的节点总数  
- `NODE_HEIGHT_ESTIMATE = 120`（来自 useSessionFrameLayout.ts 实际值，节点平均高度估算值；注释标注为 fallback，但仍是 Y 定位的唯一估算依据）

### 为何检测在 renderer 端

`usePlanIndicator.ts`（renderer）已实现：
1. 监听 `HOOK_EVENT`（PostToolUse）→ 检测 plan 文件写入
2. 调用 `IPC.PLAN_READ` 重新解析 `overall_plan.md`
3. 更新 `planNodesByProjectAtom(projectId)`

Milestone 检测只需在步骤 2→3 之间做"状态 diff"即可，与倒三角指示器共享同一数据管道，**零额外 IPC 往返**。

### 原子目标 1 — Milestone 类型定义 [x]

**文件**：`src/shared/types/index.ts`（追加，不新建文件）

```typescript
// ════════════════════════════════════════════════════════════
// 11. Milestone（任务里程碑标记）
// ════════════════════════════════════════════════════════════

export interface Milestone {
  /** PlanNode.id，如 'M4-S3-T10' */
  id: string
  /** 所属项目 ID */
  projectId: string
  /** 触发时的活跃 session claudeId（用于确定显示在哪个 SessionFrameNode） */
  sessionId: string
  /** 创建时刻 timelineBySessionAtom 的节点总数（用于 Y 坐标 nodeIndex × NODE_HEIGHT_ESTIMATE 估算） */
  nodeIndex: number
  /** 完成时间戳（ms） */
  completedAt: number
  /** 任务描述（PlanNode.title，用于 hover tooltip） */
  description: string
}
```

### 原子目标 3 — usePlanIndicator.ts 扩展：T 节点状态 diff [x]

**文件**：`src/renderer/src/hooks/usePlanIndicator.ts`

**改动位置**：`loadPlanForProject` 调用后，在 `store.set(planNodesByProjectAtom(...))` 写入新数据时做 diff。

具体逻辑：
1. 在 `usePlanIndicator` 内部维护 `prevTStatusMap = useRef<Map<projectId, Map<planNodeId, PlanStatus>>>(new Map())`，记录各项目上次加载的 T 节点状态
2. 每次 `loadPlanForProject` 完成后，对比旧 T 状态与新 T 状态：`prevStatus !== 'DONE' && newStatus === 'DONE'` → 新完成
3. 找到活跃 session：从 `hookEvent.sessionId`（PostToolUse 事件携带，即 claudeId）直接获取；若是启动时初始加载无 hookEvent，跳过（只在运行时动态创建）
4. 获取当前时间线节点数：`store.get(timelineBySessionAtom(sessionId)).length`
5. 创建 Milestone 并追加到 `milestonesByProjectAtom(projectId)`

**需要新增的 import**：
```typescript
import { milestonesByProjectAtom } from '../atoms/projects.atom'
import { timelineBySessionAtom } from '../atoms/timeline.atom'
import type { Milestone } from '@shared/types/index'
```

**注意**：`prevTStatusMap` 的 key 使用 `projectId`，内层 Map key 使用 `planNodeId`（如 `'M4-S3-T10'`）

### 原子目标 4 — SessionFrameNode.tsx 渲染 Milestone badge [x]

**文件**：`src/renderer/src/features/project-monitor/canvas/SessionFrameNode.tsx`

改动：
1. 在组件顶部读取 milestones：
   ```tsx
   import { milestonesByProjectAtom } from '../../../atoms/projects.atom'
   import { FRAME_HEADER_HEIGHT, NODE_HEIGHT_ESTIMATE } from '../../../hooks/useSessionFrameLayout'
   
   const TIMELINE_PADDING_TOP = 16   // 与 ProcessTimeline.css .ptl-container padding-top 同步

     return (
       <div
         key={m.id}
         className="sfn-milestone-badge"
         style={{ top: topPx }}
         title={m.description}
       >
         [{m.id}]
       </div>
     )
   })}

  color: var(--pu);
  border: 1px solid var(--pu);
  /* ... */
}
```

## 验收标准

- [ ] 执行 Plan 中某个 T 级任务完成后，对应 Session 框左侧立即出现紫色 `[M4-S3-T10]` badge
- [ ] badge 与进程线纵向对齐（约与触发时刻的最新节点等高，误差在 ±1 个节点高度内）
- [ ] hover badge 显示该任务完整描述文字
- [ ] 多个 T 任务依次完成 → 多个 badge 堆叠，Y 轴递增不重叠
- [ ] badge 不遮挡 ProcessTimeline 内容（left:4px，远离正文区 770px 起始位置）
- [ ] 未完成的 T 任务重新执行（`[x]` → 恢复为 `[ ]`）时不增加新 badge（diff 只响应 → DONE 方向）

## 废弃的旧版设计

### 废弃：PlanSyncService（原子目标 2）

**原计划**：`src/main/services/PlanSyncService.ts`，订阅 PostToolUse Hook，广播 `MilestoneReachedEvent`

**废弃原因**：main 进程无 `services/` 目录（实际架构未落地），且 plan 解析链路全在 renderer 的 `usePlanIndicator.ts`，无需在 main 进程重复建立。

### 废弃：MILESTONE_REACHED IPC 通道

**原计划**：`MILESTONE_REACHED (push): { sessionId, milestone: Milestone }`

**废弃原因**：检测移到 renderer 端后，数据无需跨进程传输，直接写 Jotai atom 即可。

### 废弃：三列布局

**原计划**：框内布局改为 `[里程碑列 20px][时间线内容 flex:1][右侧操作区 16px]`

**废弃原因**：SessionFrameNode 当前结构（header + sfn-timeline-area + footer）工作良好，拉动条（GlobalHistoryScrubber）在 canvas 层而非框内。改三列布局会影响 SubagentBlock 的 overflow:visible 和 BranchEdge 的 handle 定位，风险高收益低。绝对定位叠加方案等效且无副作用。

### 废弃：MilestoneMarker.tsx 独立组件

**原计划**：`src/renderer/src/features/project-monitor/MilestoneMarker.tsx`

**废弃原因**：badge 渲染逻辑极简（单个 div + 一条 CSS），inline 在 SessionFrameNode 内即可，无需独立文件增加模块数量。
