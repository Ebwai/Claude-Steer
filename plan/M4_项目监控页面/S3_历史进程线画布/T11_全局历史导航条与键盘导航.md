# T11 — 全局历史导航条与键盘导航

## 需求概述

将现有 per-frame `HistoryScrubber`（每个 SessionFrameNode 内各有一条）改为**单条全局导航条**，固定在画布右侧边缘。

两个独立的使用场景：
1. **精细化查看**：键盘 ↑↓ 逐节点跳转（覆盖 user_input / assistant / 插入线所有节点类型），视口微平移跟随，游标 overlay 指示当前位置
2. **快速定位**：鼠标拖拽导航条滑块，像浏览器滚动条一样快速跳到大概区间（按 user_input 节点分区间，鼠标操作不需要精细到每个节点）

键盘 ← → 在框之间跳转（cluster 内小跳 / 跨 cluster 大跳）。

---

## 技术背景

### 现有架构（需改造的部分）

- `HistoryScrubber.tsx`：per-session 滚动条，绝对定位于 `SessionFrameNode` 内部右侧 16px
- `scrubberIndexAtom(sessionId)`：atomFamily，per-session 当前节点索引（之前只覆盖 user_input）
- `SessionFrameNode.tsx`：实例化 `HistoryScrubber`，传入 `userInputNodes` 和 `onJumpTo`

### 现有可复用的能力

- `focusSession(sessionId)`（`useProcessLineViewport`）：让画布 fitView 到指定框，可直接复用于左右跳转视口跳转
- `focusRequestAtom`：外部触发视口跳转的解耦 atom
- `useSessionFrameLayout` 的 cluster 结构：`leftChildren` / `rightChildren` 天然描述左右导航关系
- `timelineBySessionAtom(sessionId)`：获取节点列表（包含所有类型节点）
- `ptl-container` scrollTop 机制：已实现滚动到指定节点

### 节点类型分层

```
「可键盘跳转节点」（精细化）：所有节点类型
  - user_input：用户输入节点
  - assistant：Claude 回复节点
  - insertion：所有插入线类型（branch / subagent / btw / git / insight 等）

「导航条区间点」（鼠标快速定位）：仅 user_input
  - 每个 user_input 对应导航条上一个分隔点（tick mark）
  - 鼠标拖拽定位到最近 user_input 区间即可，不需要精细到每个节点
```

### Y 轴坐标体系

进程线画布使用混合坐标系：
- @xyflow 画布坐标：框的 `x, y`（由 `useSessionFrameLayout` 计算）
- CSS scroll 坐标：每个框内的 `ptl-container.scrollTop`（节点在框内的相对位置）

全局导航条需要同时处理两套坐标：
- 纵向（上下键）：操作 CSS scroll，视口微跟随（画布 pan，不是 fitView）
- 横向（左右键）：调用 `focusSession`（`fitView` 到目标框）

## 导航逻辑设计

### 上下键（框内节点游标，精细化跳转）

```
当前聚焦框 = focusedSessionIdAtom 对应的 session
当前游标索引 = cursorNodeIndexAtom(focusedId)  ← 新 atom，覆盖所有节点

跳转节点列表 = timelineBySessionAtom(focusedId) 全部节点
  + lineInsertionsBySessionAtom(focusedId) 转换为虚拟节点
  按时间顺序排列为统一列表（jumpableNodes）

↓  →  cursor = min(cursor + 1, jumpableNodes.length - 1)
↑  →  cursor = max(cursor - 1, 0)

跳转效果：
  1. 更新 cursorNodeIndexAtom → 通过 nodeJumpRequestAtom 触发 scrollTop 跳转
  2. 画布微平移：调用 @xyflow setViewport 补偿 ΔY（见原子目标 5）
```

**jumpableNodes 排序规则**：
- timeline 节点（user_input / assistant）按其渲染顺序（timelineBySessionAtom 原有顺序）
- insertion 节点（lineInsertionsBySessionAtom）按 `timestamp` 字段插入到对应位置
- 合并后整体按时间序排列

### 左右键（框间跳转）

#### 导航规则（基于 cluster 结构）

```
当前框 = focusedSessionId
当前框类型：
  A. 是 cluster 的 parent（主线框/独立 agent 框）
  B. 是 branch（在某 cluster 的 leftChildren 或 rightChildren 中）

←（向左）：
  A → parent：当前 cluster 有左侧 branch（rel.side === 'left'）
        有 → 小跳：跳到 branchIndex 最大的左侧 branch
        无 → 大跳：跳到左侧相邻 cluster 的 parent（X 坐标更小）
  B → branch（side=left）：小跳：跳回 parent
  B → branch（side=right）：先看 parent 左侧有无 branch，有则小跳，无则大跳左侧 cluster parent

→（向右）：
  A → parent：当前 cluster 有右侧 branch（rel.side === 'right'）
        有 → 小跳：跳到 branchIndex 最大的右侧 branch
        无 → 大跳：跳到右侧相邻 cluster 的 parent（X 坐标更大）
  B → branch（side=right）：小跳：跳回 parent
  B → branch（side=left）：先看 parent 右侧有无 branch，有则小跳，无则大跳右侧 cluster parent
```

**原则**：同 cluster 内 parent ↔ branch 优先（小跳），跨 cluster 一定先落在 parent（大跳不经过对侧 branch）。

跳转后：`focusSession(targetId)` 复用现有 fitView 能力。

## 全局导航条 UI 设计

### 位置

固定在整个画布容器的最右侧，宽 16px，高 = 画布容器高度（`100%`），`position: absolute; right: 0; top: 0; z-index: 20`。

### 外观（浏览器滚动条风格）

```
┌──┐  ← 轨道（深色半透明背景，始终可见）
│  │
│██│  ← 滑块 thumb（高度 = max(40px, 1/总区间数 * 轨道高度)）
│  │     颜色：当前聚焦 session 的 agent 颜色（或默认绿 #35C98A）
│  │     拖拽时：opacity 加深
│  │
│··│  ← user_input 区间 tick marks（细小横线，2px 高，8px 宽居中）
│  │     标记每个 user_input 的相对位置（视觉参考，不可点击）
│  │
└──┘
```

**设计原则**：
- 不显示每个节点的点（与原 per-frame 滚动条的"节点点"不同）
- 只显示 user_input 的 tick marks 作为区间参考（类似浏览器标记查找结果的小线）
- 滑块代表"当前浏览位置"（即游标节点在全部内容中的相对位置）
- 多 session 时，轨道分段：每段对应一个 session，比例按各 session 的节点总数分配；段间有 2px 颜色分隔线

### 滑块（thumb）位置计算

```
轨道总节点数 = Σ(所有 session 的 jumpableNodes.length)
滑块在轨道中的 Y% = 当前游标在全局节点列表中的偏移 / 总节点数
```

### 交互

- **拖拽滑块**：按 user_input 区间粒度跳转（不按每个节点，保持鼠标操作的快速性）
  - 拖拽时按比例换算到最近的 user_input 节点，更新 scrubberIndexAtom（user_input 维度）
  - 同时触发 scrollTop 跳转（通过 nodeJumpRequestAtom）
- **点击轨道空白区**：跳到点击位置最近的 user_input 节点（同拖拽逻辑）
- **点击不同 session 段**：切换 focusedSessionIdAtom + focusSession()

### 原子目标 5 — `useGlobalKeyNav`：↑↓ 游标移动 + 视口微平移 [x]

**文件**：`src/renderer/src/hooks/useGlobalKeyNav.ts`（新建）

```typescript
export function useGlobalKeyNav(
  rf: ReactFlowInstance,           // 用于 setViewport / getViewport
): void
```

**键盘 ↑↓ 逻辑**：

```typescript
const focusedId = store.get(focusedSessionIdAtom)
if (!focusedId) return

const timelineNodes = store.get(timelineBySessionAtom(focusedId))
const insertions = store.get(lineInsertionsBySessionAtom(focusedId))
const jumpable = buildJumpableNodes(timelineNodes, insertions)

const cur = store.get(cursorNodeIndexAtom(focusedId))
const next = key === 'ArrowDown'
  ? Math.min(cur < 0 ? 0 : cur + 1, jumpable.length - 1)
  : Math.max(cur < 0 ? jumpable.length - 1 : cur - 1, 0)

store.set(cursorNodeIndexAtom(focusedId), next)
const target = jumpable[next]
store.set(nodeJumpRequestAtom, { sessionId: focusedId, nodeId: target.id, ts: Date.now() })

// 视口微平移（requestAnimationFrame 后执行，等 scrollTop 更新）
// 读取目标节点 DOM 元素的 getBoundingClientRect().top（画布容器坐标系）
// 与期望位置（视窗中心 Y）做差，调用 rf.setViewport 补偿
```

**视口微平移实现**：

```typescript
// 游标节点的 DOM 元素通过 data-node-id 属性查找（ProcessTimeline 渲染时需添加此属性）
requestAnimationFrame(() => {
  const el = document.querySelector(`[data-node-id="${target.id}"]`)
  if (!el) return
  const rect = el.getBoundingClientRect()
  const canvasRect = canvasContainerRef.current?.getBoundingClientRect()
  if (!canvasRect) return
  // 目标：节点中心对齐画布容器中心 Y
  const desiredScreenY = canvasRect.top + canvasRect.height / 2
  const actualScreenY = rect.top + rect.height / 2
  const deltaScreenY = actualScreenY - desiredScreenY
  const vp = rf.getViewport()
  rf.setViewport({ ...vp, y: vp.y - deltaScreenY }, { duration: 120 })
})
```

## 遇到的问题与修复（保留排查记录）

### [!] 问题 1：点击项目监控面板黑屏

**现象**：T11 代码合入后，点击项目监控面板立即黑屏。

**原因**：`ProcessLineCanvas.tsx` 中 `orderedSessionIds` 的 `useMemo` 被放在了两个早期 `return` 语句之后（原行 ~242）。React Rules of Hooks 要求 hook 在每次渲染中以相同顺序调用，提前 return 导致该 hook 被跳过，hook 链条数量不一致触发 React 崩溃。

**修复**：将 `orderedSessionIds` 的 `useMemo`（及所有新增 hook）移到所有条件 return 之前（行 ~160）。

**来源**：2026-04-24 实测验证

---

### [!] 问题 2：cluster 间 ←/→ 跳转完全无反应

**现象**：主线 ↔ 其 branch 之间小跳正常，但按"向外"方向键跳跨 cluster 时完全无反应。

**原因**：`useGlobalKeyNav.ts` 的 sibling 查找逻辑用了 `filter(([, r]) => ...)` 省略 key，自身 currentId 被包含进结果集（self-inclusion）。结果集长度为 1 时返回自身，`focusSession(sameId)` 不改变 atom 状态 → silent no-op。

**修复**：branch 向外侧方向按键时直接大跳到相邻 cluster parent，跳过 sibling 查找；parent 向某侧有 branch 时改为取 `children[0]`（branchIndex 最小，最近的 branch）。

**来源**：2026-04-25 实测验证
