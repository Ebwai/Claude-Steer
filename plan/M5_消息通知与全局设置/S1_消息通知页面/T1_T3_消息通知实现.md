# T1-T3 — 消息通知页面实现（合并）

## 技术原理

消息通知页面是底部标签栏的第三个 tab（`💬 消息通知 [N]`），统一处理来自各 Agent 的权限请求。

### 数据层复用（不新增 IPC）

- **`permissionRequestsAtom`**（`atoms/permission.atom.ts`）：已有完整链路
  - 写入：`permissionHandler.ts` → `enqueueRequest()` 在收到 `PermissionRequest` Hook 时追加
  - 消费：`RequestApprovalPanel`（项目监控页层③）+ 本页面（M5）——两处同时订阅，**共享同一个 atom，无需新管道**
  - 清除：`dequeueRequest(store, requestId)` —— 任意一处审批后，两处 UI 同步消失
- **`IPC.PERMISSION_RESPOND`**：已实现，`ptySessionId` + `approved` + `message?` → `ptyManager.writeToSession()`
- **链路确认**：同意 → `writeToSession(ptySessionId, 'y')` → PTY stdin → Claude Code 继续执行，**完全可行**

### 布局

```
┌───────────────────────────┬──────────────────────────┐
│ 左侧：请求列表（240px）     │ 右侧：详情面板            │
│                           │                          │
│ ▼ 项目名（折叠）           │ Agent名称 请求执行：       │
│   ● Agent (a1b2c3)        │ Bash: rm -rf ./temp      │
│     Bash 请求              │                          │
│   ● 主线程                 │ [同意]  [拒绝]            │
│     Write 请求              │                          │
│                           │ 附加信息（可选）：          │
│ （FIFO 时间顺序）           │ [___________________]    │
│                           │ [发送]                   │
└───────────────────────────┴──────────────────────────┘
```

### 桌面通知架构（主进程发出）

系统级桌面通知必须在**主进程**用 Electron `Notification` API 发出，不用 Web `Notification` API（渲染进程的 Web Notification 在 Electron 中行为不一致）。

触发时机：`HookEventBus.ts` 收到 `PermissionRequest` 事件时，调用 `NotificationService.notify()`。

**三平台实现差异**：

| 平台 | 桌面通知 | 任务栏角标 |
|------|---------|----------|
| Linux（Ubuntu/GNOME）| `new Notification({ title, body }).show()` + `Notification.isSupported()` 前置检查 | `app.setBadgeCount(N)`（Unity/GNOME Dock 支持）|
| macOS | `new Notification({ title, body }).show()` | `app.dock.setBadge(String(N))`（N=0 时传 `''` 清除）|
| Windows | `new Notification({ title, body }).show()` | `mainWindow.setOverlayIcon(icon, label)`（`setBadgeCount` 在 Windows 无效）；N=0 时 `setOverlayIcon(null, '')` 清除 |

> ⚠️ Important_Info 已记录：Windows `setBadgeCount` 不支持数字，必须用 `setOverlayIcon`。

## 实现步骤

### 原子目标 T1 — 消息通知页面骨架 + 左侧请求列表 [ ]

**文件**：
- `src/renderer/src/features/notifications/NotificationsPage.tsx`
- `src/renderer/src/features/notifications/NotificationList.tsx`
- `src/renderer/src/features/notifications/NotificationsPage.css`

**NotificationsPage.tsx**：
- 左右两列布局（`display: grid; grid-template-columns: 240px 1fr`）
- 左侧挂载 `NotificationList`，右侧挂载 `NotificationDetail`（无选中时显示空状态）
- 本地 `useState<string | null>(selectedRequestId)` 控制当前选中

**NotificationList.tsx**：
- `useAtomValue(permissionRequestsAtom)` 订阅待处理队列
- FIFO 排序：按 `receivedAt` 升序
- 按 `sessionId` 分组，组头显示 `agentName`（如 `Agent (a1b2c3)` 或 `主线程`）
- 每项显示：`● toolName` + `description` 前 40 字
- 选中项高亮（`--or-500` 左边框），点击更新 `selectedRequestId`
- 无请求时：「暂无待处理请求」空状态居中显示
- 日志：`console.log('[Notif] list rendered, count=', requests.length)`

### 原子目标 T2 — 右侧详情面板 + 审批操作 [ ]

**文件**：`src/renderer/src/features/notifications/NotificationDetail.tsx`

- props：`requestId: string | null`，内部从 `permissionRequestsAtom` 查找对应条目
- 标题：`[agentName] 请求执行：`
- 操作描述块：`toolName` 大字 + `description` 次字 + `toolInput` JSON 折叠展示（`<details>`）
- 操作按钮行：
  - **「同意」**（绿色）：直接调用 `handleRespond(true, '')`
  - **「同意并附加信息」**（蓝色）：展开附加信息输入框
  - **「拒绝」**（红色）：直接调用 `handleRespond(false, '')`
- 附加信息输入框（仅展开时显示）：`textarea` + Enter 发送（`Shift+Enter` 换行）+ 「发送」按钮
- `handleRespond(approved, message)` 逻辑：
  ```
  1. await window.api.invoke(IPC.PERMISSION_RESPOND, { sessionId, ptySessionId, approved, message })
  2. dequeueRequest(store, requestId)   // 清除 atom → 两处 UI 同步消失
  3. console.log('[Notif] respond ok:', requestId, approved)
  ```
- 错误时：inline 红字提示，不清除请求（允许重试）
- `requestId` 为 null 时：显示「← 从左侧选择请求」

### 原子目标 T3 — 桌面通知 + 任务栏角标（主进程 NotificationService）[ ]

**文件**：`src/main/lib/notification/NotificationService.ts`

```
NotificationService
  ├── notify(title, body)   → Notification.isSupported() 检查 → new Notification().show()
  ├── setBadge(n, win)      → platform 分支：
  │     macOS:   app.dock.setBadge(n > 0 ? String(n) : '')
  │     Linux:   app.setBadgeCount(n)
  │     Windows: win.setOverlayIcon(n > 0 ? badgeIcon : null, n > 0 ? `${n} 待处理` : '')
  └── badgeIcon             → nativeImage.createFromDataURL(RED_DOT_BASE64)（16×16 红圆点）
```

**HookEventBus.ts 触发点**（在已有 PermissionRequest 分支内追加）：
```typescript
// 已有代码：webContents.send(IPC.HOOK_EVENT, hookEvent)
// 追加：
if (payload.hook_event_name === 'PermissionRequest') {
  const desc = (payload as { description?: string }).description ?? payload.tool_name ?? '操作'
  NotificationService.notify('Claude Driver', `[${agentLabel}] 请求执行：${desc}`)
}
```

**角标更新时机**：
- `index.ts` 中监听 `IPC.PERMISSION_RESPOND` 成功后：`pendingCount--` → `NotificationService.setBadge(pendingCount, mainWindow)`
- `HookEventBus` 收到新 PermissionRequest 时：`pendingCount++` → `NotificationService.setBadge(pendingCount, mainWindow)`
- 主进程维护 `let pendingPermissionCount = 0`（模块级变量）

**Windows badge 图标资源**：
- 用 `nativeImage.createFromDataURL()` 内联一个 16×16 红圆点 Base64 PNG，不依赖外部文件

**日志**：
```
[NotifSvc] notify: title=... body=...
[NotifSvc] setBadge: n=N platform=linux/mac/win
[NotifSvc] Notification.isSupported=false, skip
```

## 验收标准

- Claude 触发 `PermissionRequest` Hook：
  - 消息通知 tab 角标 +1
  - 桌面通知弹出（标题 + 操作描述）
  - 底栏「N 待处理请求」+1
- 消息通知页面左侧列表显示请求（含 agentName + toolName）
- 选中请求后右侧显示详情（toolInput 可展开查看）
- 点击「同意」：PTY stdin 写入 `y`，Claude 继续，列表项消失，角标 -1
- 点击「拒绝」：PTY stdin 写入 `n`，列表项消失，角标 -1
- 项目监控页面层③审批框同步消失（共享 atom）
- Ubuntu 上桌面通知正常弹出（libnotify）
- Windows 任务栏显示红点覆盖图标（不显示数字）
- macOS Dock 图标显示数字角标
