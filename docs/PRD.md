## 0. 底层设计哲学

Claude Steer 的所有产品设计围绕一个核心理念展开：**围绕信息的设计**。这不是一个功能堆砌的工具，而是一个对"信息"进行全生命周期管理的系统。理解这一哲学是理解后续所有概念与机制的前提。
【对应产品设计的这列可以删除】
| 信息维度 | 设计内涵 | 对应产品设计 |
|---------|---------|------------|
| **信息的产生（捕获）** | 捕获 Agent 工作过程中的全部数据；接入高质量信息源的推送 | 三通道数据采集（Hooks/statusLine/JSONL）、高质量信息源推送（作者推荐） | 
| **信息的输出（展示）** | 按场景、按需求进行精细颗粒度显示，只展示当前场景下值得展示的信息，同时支持不同颗粒度的查看；处理信息与人的注意力的关系 | 渐进式信息披露、信息密度可调节的可视化、十类插入线分层 |
| **信息的保存** | 项目历史面板、运行日志全留存，用于问题根因回溯、AI 幻觉事前核验+事后排查整套流程 | 历史工作区、上下文面板、JSONL 全量留存 |
| **信息的管理** | 配置项的直观精细化管理；信息的沉淀与量化提炼；基于多 Agent 的交互设计 | 全局设置、项目设置栏、深度搜集机制、增量原子写入、[规划中]工作知识库 |
| **信息的连接** | 基于 cc-connect 实现远程交互信息可视化 | 远程交互功能入口 |

**衍生设计原则**（贯穿所有概念）：
1. **第一性原理分工**：每个功能处理它最擅长的事——CLAUDE.md 处理始终开启的上下文，Skills 处理按需知识，MCP 处理外部连接，Subagents 处理隔离执行，Hooks 处理自动化
2. **渐进式信息披露**：每个场景下只显示当前所需的必要信息，详情放入子页面/可展开区域
3. **完全兼容官方**：遵循 Claude Code 官方最佳实践，不重造 Claude Code，而是封装与增强其能力
4. **跨平台**：Windows / macOS / Ubuntu 三平台一致体验
5. **AI Native 设计**：所有可视化和交互设计理念基于 Agent 原生，软件架构是适合于Agent自开发的原生设计，开发流程是为Agent设计的。是区别于基于 AI 前时代 IDE 理念改造的竞品

### 2.1 详细说明

#### 概念一：全局监控界面

> 全局统览的信息视图和配置入口。用户打开软件后首先看到的界面，承担"项目总览 + 全局配置 + 功能入口"三大职责。左右分栏：左半为项目画板，右半为配置面板。

##### 1.1 项目画板（无限画布）

- **功能描述**：一块可缩放、可平移的无限画布，以可视化卡片形式展示用户全部值得关注的长期项目，以及顶部一个代表用户自身的节点。进行中的项目展开显示其 M 级 Plan 进度，其他项目折叠显示。
- **业务规则与约束**：
  - 项目卡片分为"进行中"与"其他项目"两类：进行中项目默认展开至 M 级 Plan 列表（最多展示 4 条）；其他项目不展开。
  - 进行中项目卡片独有"倒三角执行指示器"（见下），用于标识当前是否有 Agent 正在该项目工作。
  - 卡片交互：单击展开/折叠下一级详情；双击切换到底部"项目监控"标签并自动打开该项目。
  - 用户节点位于画布左上角，显示头像与可编辑用户名（默认"我"）。
- **数据范围**：项目数据来自项目记录单（含项目路径、认领状态、运行状态）；Plan 数据来自每个项目对应的 `plan/overall_plan.md`。

###### 1.1.1 倒三角执行指示器

- **功能描述**：进行中项目卡片上的倒三角图标，表示该项目当前有 Agent 正在工作。是一个动态状态指示器。
- **业务规则与约束**：
  - 触发条件：检测到该项目下 Agent 的工具调用或 Plan 文件变动。
  - 销毁条件一：M 任务全部完成，且一段时间内无新 Plan 文件变动 → 销毁。
  - 销毁条件二：M 任务未完成，但一段时间内无 Plan 变动 → 转为"可能暂停"状态。
  - 支持多 Plan 并行场景。
- **数据范围**：基于工具调用事件与 Plan 文件变动监听。

###### 1.1.2 新建项目入口 / 创建项目向导

- **功能描述**：从画布新建一个长期项目，通过三步向导完成项目初始化。
- **业务规则与约束**（三步流程）：
  - **Step 1·项目设置**：输入项目名称（加入项目记录单）；以按钮形式选择该项目可调用的 Skills/MCP/Tools/Agent/Plugins；选择权限模式（6 选 1，悬停显示说明）；可选展开系统提示词配置界面（支持模板变量填充）。
  - **Step 2·放入资产**：提示用户将调研报告、demo 代码等资料放入项目资料文件夹；点击"已放好"继续。
  - **Step 3·制定计划**：自动跳转到项目监控页面；自动在项目文件夹打开新 Claude Session（权限默认设为 acceptEdits）；弹出 Claude 终端子窗口；自动发送"创建计划"指令。
- **数据范围**：写入项目记录单；写入项目级 `.claude/settings.json`；启动 PTY 会话。

###### 1.1.3 闲聊入口

- **功能描述**：对使用 Claude Code 场景的补充——需要长时间持续工作的任务通过创建项目，临时问一些问题或流程不长的小任务通过闲聊。点击后弹出独立的 Claude 终端子窗口（非项目绑定）。
- **业务规则与约束**：闲聊产生的 token 归入全局统计的"未分类"项，不归属任何项目。
- **数据范围**：独立终端会话；token 统计。

###### 1.1.4 Plugins 面板 `[已实现]`

- **功能描述**：管理 Claude Code Plugins（插件集合，一个 Plugin 可包含 MCP/Skills/Hooks/Agents）。原稿未提及此概念，但代码已实现。
- **业务规则与约束**：
  - 显示已安装 Plugin 列表（读取 `settings.json` 的 `enabledPlugins` 字段），每个 Plugin 可查看详情（应用场景、作用、包含的 MCP/Skills/Hooks）。
  - 添加 Plugin：第一阶段显示"开发中"占位 `[部分实现]`。
  - 精品推荐区：预留空位 `[规划中]`。
- **数据范围**：`~/.claude/settings.json` 的 `enabledPlugins` 字段。

##### 1.2 全局统计

- **功能描述**：三等分网格卡片，展示"最常用模型 / 本月 Token 用量 / 累计费用"三项全局指标，让用户一眼掌握 AI 使用成本。
- **业务规则与约束**：
  - **最常用模型**：根据 token 消耗量统计用户使用最多的模型。
  - **本月 Token 用量**：对应流行的月度 coding plan 计划。**当前只支持不区分 provider 的统计**（未来会优化按 provider 区分）`[部分实现]`。
  - **累计费用**：有对全部花销的统计，以及对每个长期项目的 AI 花销统计（价格可自定义）。
  - 费用计算：基于 token 用量与用户自定义的单价计算。
  - 统计归属规则：全局总计 = 所有项目之和 + "未分类"（闲聊归入未分类）。

##### 1.4 Agent 工具和经验

> **为什么这么划分？这是对人类员工的抽象**：大模型类比于人自身的认知与基础常识逻辑推理能力；工具就是其在所在行业经常使用的工具链（站在前人肩膀上简化工作量）；经验就是长期工作中试错-沉淀得出的针对特定任务场景的相对较优解法思路。

- **功能描述**：集中展示与管理当前可被 Agent 使用的"工具"与"经验"两类资源，让用户直观看到有哪些可用、从哪来、如何添加。包含 Agent 面板、经验面板（Skills + 工作流）、工具面板（Tools/MCP/CLI）。

##### 2.3 历史工作区

> 单个 Agent 的工作进程框，完整还原从指令输入、思考推理、落地执行到多体协作的全流程。是**问题根因回溯与 AI 幻觉事后排查**的核心载体。基于 JSONL 转录文件全量留存。

###### 2.3.1 主线 Agent 工作框

- **功能描述**：采用「时间中轴线 + 左右分层 + 分叉扩展」的可视化结构，共划分为四类核心事件。
- **业务规则与约束**（四类核心事件）：

  **① 人与 Agent 交互节点（时间中轴线）**——整条工作链路的主干节点，类比甘蔗分节与树干节点：
  - 以 Agent 的关键阶段性回复作为核心节点标记
  - 用户向 Agent 输入指令的入口
  - 通过这些节点可快速把握整条时间线的核心推进脉络，大颗粒度定位人工介入交互的关键位置

  **② Agent 思考层事件（中轴线左侧）**——对应 Agent 的内在认知与思维碰撞过程，所有事件沿时间中轴线左侧排布，skills / 工作流 / BTW / Insight 采用不同长度的插入线标识，便于快速识别与事后复盘：
  - 经验调用：Skills 调用、工作流调用（工作流目前仅通过 Hooks 实现，未来计划融入 n8n 工作流体系——agent 与 n8n 并非竞争关系，而是不同场景下的不同选择）
  - 思维碰撞节点：
    - **BTW**：基于 Prompt Cache 机制的临时试探性提问，用于用户对 Agent 工作过程产生疑问时快速交互，充分利用缓存降低 AI 成本；单次回答不进行 action，不增加原有对话上下文
    - **Insight**：Claude Code 原生机制，记录 Agent 解决问题过程中的关键突破点与核心洞见，相当于人工执行任务时最值得复盘沉淀的关键节点

  **③ Agent 行动层事件（中轴线右侧）**——对应 Agent 将思考落地执行的具体动作，所有事件沿时间中轴线右侧排布，且 tool / MCP / CLI 采用不同长度的插入线标识，可快速区分 Agent 调用的工具类别：
  - Agent 自带工具（Tools）
  - MCP 工具
  - CLI（识别为 Skill 包中带 cli 字的）

  **④ 多 Agent 分叉事件**——从任意主线交互节点均可触发创建多 Agent 分支，依托 Claude Code 原生的 branch / subagent / BTW 三个核心概念实现：
  - **Subagent**：由主 Agent 分配独立执行目标单一、路径清晰的专项任务（如资料检索、信息整理），主 Agent 仅接收最终结果，避免中间过程的无效信息占用上下文窗口。创建时支持配置：指派某个 subagent 的目标、设定新 subagent 的人设、subagent 的数量
  - **Branch**：用于路线试错、项目答疑或并行作业三类场景，从主线节点分叉出独立探索分支，完整记录探索过程与结果，不干扰主链路推进
  - **BTW**：本质是继承主线 Agent 上下文缓存的一次性轻量 Agent，对话结束即销毁，非常适合针对主线工作内容发起临时疑问与快速验证

- **Plan Tag**：假如 Agent 完成了某项制定的 Plan 里的某条，那么会标记对应的原子目标。
- **里程碑标记**：Plan 完成后插入里程碑标记（如 `[M1-S1-T1]`），标记阶段性成果。
- **数据范围**：JSONL 转录文件（user/assistant 节点）+ 三通道事件（插入线）+ Plan 状态（Plan Tag）+ 里程碑记录。

###### 2.3.2 节点点击菜单

##### 2.4 Plan 区

- **功能描述**：长期项目的实现计划，显示当前工作进展（已完成的、未完成的、现在在进行哪个子任务目标）。
- **业务规则与约束**：
  - **三级管理**（也可自定义更多级，默认三级）：母任务(M) - 子任务(S) - 原子任务(T)。
  - **制定多级 Plan 的原因**：可以尽可能确保 Agent 不会偏离方向，每个 Plan 要有检查标准，只有通过了才可以进行下一个 Plan 的工作，这样可以保证 Agent 的工作像搭积木一样——之前搭建的积木是坚实的，再往上搭的时候假如搭错了，只会影响最上层的，下面的依然可以复用。
  - **Plan 文件格式**（**重要：与全局/项目监控界面的 Plan 面板解析逻辑强绑定，格式不正确将导致无法识别**）：

    | 层级 | 必须写成 | 示例 |
    |------|---------|------|
    | M（母目标） | `## M<数字> - <标题> [状态]` | `## M1 - 项目脚手架与基础架构层 [ ]` |
    | S（子目标） | `### S<数字> - <标题> [状态]` | `### S1 - electron-vite 项目初始化 [x]` |
    | T（任务） | `- [<状态字符>] T<数字> - <标题>` | `- [!] T2 - 实现双向 invoke/handle 机制` |

    状态字符：`[ ]` 待办 / `[x]` 已完成 / `[!]` 进行中。

  - Plan 文件路径：相对于项目根目录，统一命名 `plan/`，主文件为 `overall_plan.md`。
  - 结合 Git 功能，可以实现更高效更安全的 Agent 工作模式。
- **数据范围**：项目 `plan/` 目录下的 Markdown 文件。

##### 2.5 Git 功能

- **功能描述**：每个有文件修改的对话节点都可以手动一键 Git 和同步到 GitHub，防止 Agent 跑偏可以及时回到正确版本中。是"计划与可靠性"的工具支撑。
- **业务规则与约束**：
  - 每个 Main Session 一个独立 Git worktree 分支（映射：session_id → worktree_path → branch_name）。
  - **节点级 Git 操作**：Git 当前版本（快照）/ 回退到当前版本 / 删除节点（含 rebase）。
  - **项目级 Git 操作**：项目设置栏最右端"同步到 GitHub"按钮（全局触发推送）。
  - 可用 Git 操作：commit / reset / push / 确认仓库 / 删除 commit（非交互式 rebase，**禁止交互式 rebase**）。
  - 合并到 Main：`[占位·代码中标注 not yet implemented]`。
  - Git 远程未配置或权限不足 → 弹出子窗口说明操作步骤。
- **数据范围**：项目 worktree；Git 快照标记持久化。

### 2.2 机制详细说明

> 机制是为了实现概念而在底层运行的功能，用户没有直接接触，但保障了概念功能的正常运行。一个机制可被多个概念复用。

#### 机制一：多 Agent 点对点管理

- **功能描述**：一个统一界面对多 Agent 的点对点管理，是整个软件的底层核心机制。贯穿全局监控与项目监控。
- **业务规则与约束**：
  - **单个命令终端的识别 / 创建 / 写入 / 读取 / 生命控制**：
    - 识别：通过 Session ID 唯一标识每个 Agent 会话
    - 创建：四种启动模式（新建 / 恢复 / 裸启动 / 命令启动）
    - 写入：通过 stdin 管道向会话注入指令（含消息队列注入、权限审批注入）
    - 读取：三通道读取（Hooks 主事件流 / statusLine 实时 / JSONL 历史回看）
    - 生命控制：心跳检测（定期检测进程存活）、超时关闭（长时间无交互自动关闭）
  - **多个命令终端的统一管理**：点击回到对话、实时工作区和历史工作区的对应机制、【规划中】同项目不同 Agent 之间的切换（快速切换与精细化切换）。
  - **对话生命周期管理**：
    - 所有 Session 必须通过仪表盘启动（才能完整管控）
    - 终端子窗口可关闭不终止 PTY 进程（仅取消渲染，重新打开可恢复）
    - 手动启动的 Claude 进程：仪表盘可扫描 JSONL 转录文件只读显示，但**无法注入指令**
    - 长时间无新交互自动关闭（资源释放）+ 发出提醒通知
- **数据范围**：PTY ↔ Claude Session 双向绑定表；Session 元数据；JSONL 转录。

#### 机制二：Subagent 显示逻辑

- **功能描述**：确保多个 Subagent 添加时始终出现在 Agent 旁边，且多个 Subagent 时显示不会冲突。
- **业务规则与约束**：
  - SubagentStart Hook 触发时插入 Mini 时间线。
  - Subagent 内部工具调用归属于子 Agent 的 agent_id，主 Session Hooks 行为看不到。
  - 槽位分配机制管理多个并发 Subagent 的显示位置。
- **数据范围**：SubagentStart/SubagentStop Hook 事件；Subagent JSONL 转录。

#### 机制三：Branch 显示逻辑

- **功能描述**：确保多个 Branch 添加时始终出现在 Agent 旁边，且多个 Branch 不会显示冲突，以及插入点的固定。
- **业务规则与约束**：
  - Branch 三态状态机管理（创建中 / 运行中 / 结束）。
  - 父子关系注册表维护 Branch 与主线的关联。
  - 插入点固定在主线对应节点。
- **数据范围**：Branch 父子关系注册表；Branch JSONL 转录。

#### 机制四：增量原子写入配置

- **功能描述**：只管理特定字段，不是重写文件全部内容，不会影响原有的其他部分内容。解决"修改配置项担心覆盖其他内容"的痛点。
- **业务规则与约束**：
  - 所有配置文件统一采用"写临时文件 + 原子重命名"模式，保证写入的原子性（不会出现半写状态）。
  - 字段级合并：读取原文件 → deep merge 指定字段 → 原子写回。
  - 适用所有配置：项目记录单、全局配置、CC settings.json、`~/.claude.json`、定时任务、cc-connect 配置。
- **数据范围**：全部配置文件。

#### 机制五：深度搜集机制

- **功能描述**：本身遵循的是 Claude Code 对所有 skill / mcp / tool 在真实 Agent 工作时的识别机制来设计的。Claude Steer 是读取的，所以用户自己修改对应文件里的，Claude Steer 也会自动同步更新。
- **业务规则与约束**：
  - 自己添加的或者是按照 Claude Steer 中添加的都会被自动识别到。
  - 所有渠道添加到（只要是当前版本 Claude Code 在执行中会用到的都会在 Claude Steer 中显示），包括各个 Plugin 里的。
  - 所有的配置项也是，Claude Steer 读取后用户自己修改对应文件，Claude Steer 也会自动同步更新。
  - 一次性返回 5 类配置：Agents（内置 5 种 + 用户 .md frontmatter + 插件）、Skills（用户子目录 + 插件）、Hooks、Tools（内置 15 种）、MCP（全局 + 项目级）。
- **数据范围**：`~/.claude/agents/`、`~/.claude/skills/`、`~/.claude/settings.json`、`~/.claude.json`、项目级配置文件。

#### 机制六：上下文更新机制

- **功能描述**：阅读新信息的添加，以及触发 compress 后的删除。实时追踪每个 Agent 当前上下文，支撑幻觉事前核验。
- **业务规则与约束**：
  - PostToolUse 时追踪 Read / Glob / Grep（file 类型）与 WebFetch（WebFetch 类型），添加为动态上下文组件。
  - PostCompact 时清空动态组件（非持久化组件），持久组件保留。
  - 上下文组件分持久（System Prompt / CLAUDE.md / Auto Memory / Skills）与动态（Read 文件 / WebFetch）两类。
  - **原稿备注**："其实细节还没有做的很好，最近比较忙，会在未来几次推送中不断完善" `[部分实现]`。
- **数据范围**：PostToolUse / PostCompact Hook 事件；上下文组件列表。

#### 机制七：Plan 显示与自动跟随

- **功能描述**：Plan 的显示是基于特定文件的读取，所以只要有文件且符合特定格式就可以显示且可随时更新。会不断扫描 Plan 的更新情况反应在 Plan 当前执行状态的指示上。
- **业务规则与约束**：
  - **格式规则**：已设定在所有通过 Claude Steer 新建的项目中（见 Plan 区格式表）。可以直接说需求和要求即可生成可被识别并显示的属于项目的 Plan。
  - **自动跟随**：PostToolUse Hook 检测 Plan 文件写入后自动重新拉取 `overall_plan.md` 并刷新；解析 M/S/T 三级，状态 diff 检测新完成的 T 任务创建里程碑。
  - **双链路保障**：主链路 Hooks 驱动（实时）+ 兜底链路独立线程全量读取（定期轮询）修正 Hooks 漏报的差异。
  - Plan 文件损坏/格式不符 → 直接不显示该条目，不报错崩溃。
  - **注意**：原稿提到 `plan_state.json` 缓存文件，但代码中不使用独立 JSON 缓存，而是直接解析 `overall_plan.md` 的 Markdown 文本，运行时状态保存在内存中。
- **数据范围**：项目 `plan/overall_plan.md`。

## 3. 交互流程和细节

### 3.1 首次打开·初始化 SOP

**前置条件**：
- 首次启动软件，项目记录单为空

**主流程**：
1. 软件检测依赖（Node.js ≥18 / npm / Git / Claude Code CLI）→ 若缺失则引导安装（支持 Claude CLI 自动安装）
2. 引导用户选择"存在所有项目的文件夹"
3. 递归扫描全部含 `CLAUDE.md` 的文件夹（排除 node_modules 等，最大深度 6）→ 写入项目记录单并标记
4. 向用户发送项目认领清单
5. 用户对每个项目选择认领状态：`1` = 确认是项目 / `0` = 不是项目 / `-1` = 稍后决定（默认）

**反馈规则**：
- 依赖缺失 → 弹窗提示并引导
- 扫描完成 → 显示认领清单
- 认领完成 → 进入全局监控主界面

**前置条件**：

### 3.3 项目监控·实时驾驭 Agent

**前置条件**：
- 已有项目，且至少一个 Agent 在工作

**主流程**：

## 4. 边界情况与异常处理

### 4.1 边界情况

1. **Plan 文件损坏 / 格式不符**：直接不显示该条目，不报错崩溃（保证健壮性）。
2. **手动启动的 Claude 进程**：仪表盘可扫描 JSONL 转录文件只读显示，但无法注入指令（管控边界）。
3. **终端子窗口关闭**：不终止 PTY 进程，仅取消渲染输出；重新打开窗口可恢复 PTY 输出流接入。
4. **终端子窗口长时间无交互**：自动关闭（资源释放）+ 发出提醒通知。
5. **进程消失**：心跳检测发现进程消失 → 标记为"已中断"状态。
6. **多个 Agent 同时权限请求**：审批框 FIFO 堆叠，无超时。
7. **Subagent 内部工具调用可见性**：归属于 Subagent 的 agent_id，主 Session Hooks 行为看不到，需在子 Agent Hook 事件中区分。
8. **删除节点区间含 Git commit**：同步执行非交互式 rebase 删除对应历史，禁止交互式 rebase。
9. **已 Git 节点也可删除**：不保护，删除后 Git 历史同步清除。
10. **未 Git 节点回退限制**："回退到当前版本"显示灰色不可交互。
11. **非最后节点创建多 Agent 限制**："创建多 Agent"显示灰色不可交互。
12. **JSONL 重复读取**：采用 tail 追加模式（文件监听 + 定时轮询兜底），避免重复处理已读内容。
13. **Plan 状态同步兜底**：独立线程定期全量读取所有 Plan 文件，修正 Hooks 漏报差异。
14. **文件监听静默停止**（部分平台长时间运行）：配置定时轮询兜底。
15. **Git 远程未配置 / 权限不足**：弹出子窗口说明操作步骤。
16. **CLI 识别规则**：在用户的 `.claude/skills` 下识别所有带 cli 字的 skill 为 CLI，不放入 skill 类。
17. **本月 Token 不区分 provider**：当前只支持不区分 provider 的统计，按 provider 区分为未来优化方向。
18. **Hook 注册合并**：首次启动自动注入 Hook 配置，合并到已有 hooks 配置（不覆盖用户其他 hooks）。

### 4.2 异常处理

| 异常场景 | 处理策略 |
|---------|---------|
| 依赖缺失（Node/npm/Git/Claude CLI） | 弹窗提示 + 引导安装（Claude CLI 支持自动安装） |
| Plan 格式错误 | 跳过该条目，不崩溃 |
| 进程异常消失 | 心跳检测标记"已中断"，用户可"回到对话"恢复 |
| Git 操作失败 | 弹出子窗口说明操作步骤 |
| cc-connect 未安装 | 检测并提示，提供 CLI 命令供手动安装 |
| /branch 检测异常 | Ctrl+C 强制退出 → SessionEnd → `claude -r` 恢复链路 |
| Session 启动失败 | 静默处理 `[待确认·是否需更明确反馈]` |
| 端口被占用 | `[待确认·具体处理方式]` |

### 4.3 降级策略

1. **Hooks 失效降级**：当 Hooks 主链路失效时，兜底链路定期全量读取 Plan 与 JSONL，保证数据不丢失。
2. **statusLine 首次调用前降级**：首次 API 调用前 context_window.current_usage 为 null，统计显示占位。
3. **手动启动进程降级**：只读显示，不提供注入能力（明确的能力边界，而非报错）。
4. **未开放功能降级**：Auto-Dream 等未开放功能点击显示友好提示，不写入配置，待官方开放后替换为真实功能。
