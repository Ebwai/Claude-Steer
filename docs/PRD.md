> **来源依据**：本文档基于已确认的 `docs/MRD.md`（Claude Steer 市场需求文档）展开，概念与机制骨架来源于 `软件介绍原稿.md`，所有交互流程、业务规则、边界情况均经过 `claude-driver/` 实际代码验证。
>
> **与旧版关系**：根目录 `PRD.md`（Claude Driver v0.2.0-draft）为历史版本，本文档为基于 Claude Steer 新定位的重写版本，保留旧文件作为历史参考。旧版中与代码/CLAUDE.md 冲突之处（如 Plan 格式 `TODO/DOING/DONE`、产品名 Claude Driver）已在本文档修正。
>
> **状态标注约定**：
> - `[已实现]`：代码中已完整实现并验证
> - `[部分实现]`：代码中有实现但不完整或有占位

## 0. 底层设计哲学

Claude Steer 的所有产品设计围绕一个核心理念展开：**围绕信息的设计**。这不是一个功能堆砌的工具，而是一个对"信息"进行全生命周期管理的系统。理解这一哲学是理解后续所有概念与机制的前提。
| 信息维度 | 设计内涵 |
|---------|---------|
| **信息的产生（捕获）** | 捕获 Agent 工作过程中的全部数据；接入高质量信息源的推送 |
| **信息的输出（展示）** | 按场景、按需求进行精细颗粒度显示，只展示当前场景下值得展示的信息，同时支持不同颗粒度的查看；处理信息与人的注意力的关系 |
| **信息的保存** | 项目历史面板、运行日志全留存，用于问题根因回溯、AI 幻觉事前核验+事后排查整套流程 |
| **信息的管理** | 配置项的直观精细化管理；信息的沉淀与量化提炼；基于多 Agent 的交互设计 |
| **信息的连接** | 基于 cc-connect 实现远程交互信息可视化 |

**衍生设计原则**（贯穿所有概念）：
1. **第一性原理分工**：每个功能处理它最擅长的事——CLAUDE.md 处理始终开启的上下文，Skills 处理按需知识，MCP 处理外部连接，Subagents 处理隔离执行，Hooks 处理自动化
2. **渐进式信息披露**：每个场景下只显示当前所需的必要信息，详情放入子页面/可展开区域
3. **完全兼容官方**：遵循 Claude Code 官方最佳实践，不重造 Claude Code，而是封装与增强其能力
4. **跨平台**：Windows / macOS / Ubuntu 三平台一致体验
5. **AI Native 设计**：所有可视化和交互设计理念基于 Agent 原生，软件架构是适合于Agent自开发的原生设计，开发流程是为Agent设计的。是区别于基于 AI 前时代 IDE 理念改造的竞品

### 2.1 详细说明

#### 概念一：全局监控界面

> 全局统览的信息视图和配置入口。用户打开软件后首先看到的界面，承担"项目总览 + 全局配置 + 功能入口"三大职责。左右分栏：左半为项目画板，右半为配置面板。

##### 1.1 项目画板（无限画布）

- **功能描述**：一块可缩放、可平移的无限画布，以可视化卡片形式展示用户全部值得关注的长期项目，以及顶部一个代表用户自身的节点。进行中的项目展开显示其 M 级 Plan 进度，其他项目折叠显示。
- **业务规则与约束**：
  - 项目卡片分为"进行中"与"其他项目"两类：进行中项目默认展开至 M 级 Plan 列表（最多展示 4 条）；其他项目不展开。
  - 进行中项目卡片独有"倒三角执行指示器"（见下），用于标识当前是否有 Agent 正在该项目工作。
  - 卡片交互：单击展开/折叠下一级详情；双击切换到底部"项目监控"标签并自动打开该项目。
  - 用户节点位于画布左上角，显示头像与可编辑用户名（默认"我"）。
- **数据范围**：项目数据来自项目记录单（含项目路径、认领状态、运行状态）；Plan 数据来自每个项目对应的 `plan/overall_plan.md`。

###### 1.1.1 倒三角执行指示器

- **功能描述**：进行中项目卡片上的倒三角图标，表示该项目当前有 Agent 正在工作。是一个动态状态指示器，三态：active（工作中）/ possibly-paused（可能暂停）/ completed（已完成）。
- **业务规则与约束**：
  - **触发（active）**：检测到 Plan 文件写入（PostToolUse 的 Write/Edit/MultiEdit 且路径匹配 `plan/*.md`）。
  - **转暂停（possibly-paused）**：active 状态下 5 分钟内无新 Plan 文件变动。
  - **销毁（completed）**：M 任务全部完成（所有 M 节点 DONE）后 3 分钟清空指示器。
  - 支持多 Plan 并行场景。
- **数据范围**：Plan 文件变动监听（PostToolUse Hook）；Plan 状态来自 `plan/overall_plan.md`。
- **相关代码**：`renderer/src/hooks/usePlanIndicator.ts`（触发/超时/销毁逻辑）、`renderer/src/features/global-monitor/nodes/ProjectCardNode.tsx`（倒三角渲染与三态样式）。
- **实现偏差 `[待确认]`**：PRD 原写"工具调用**或** Plan 文件变动"两个并列触发条件，但代码实际只接受 Plan 文件写入（PostToolUse -> Write/Edit/MultiEdit -> `plan/*.md`），任意工具调用（Bash/Read 等）不触发。即"工具调用"只是检测链路的一环，并非独立触发源。供后续核对实际运行效果。

###### 1.1.2 新建项目入口 / 创建项目向导
- **功能描述**：从画布新建一个长期项目，通过三步向导完成项目初始化。
- **业务规则与约束**（三步流程）：
  - **Step 1·项目设置**：输入项目名称（加入项目记录单）；选择权限模式（6 选 1，悬停显示说明）。
    - **`[待开发]`**：原稿描述的"以按钮形式选择可调用的 Skills/MCP/Tools/Agent/Plugins"与"系统提示词配置界面（含模板变量 `{{PROJECT_NAME}}` 等）"在代码中均待开发（`systemPrompt`/模板变量在全仓零匹配；Step 1 代码未见 Skills 选择按钮）。向导 Step 1 实际仅含：名称 / 路径 / 描述 / 权限。
    - **相关代码**：`renderer/src/features/global-monitor/CreateProjectWizard.tsx`（`WizardStep = 1 | 2 | 3`，Step 1 实际为名称/路径/描述/权限）。
  - **Step 2·放入资产**：提示用户将调研报告、demo 代码等资料放入项目资料文件夹；点击"已放好"继续。
  - **Step 3·制定计划**：自动跳转到项目监控页面；自动在项目文件夹打开新 Claude Session（权限默认设为 acceptEdits）；弹出 Claude 终端子窗口；自动发送"创建计划"指令。
- **数据范围**：写入项目记录单；写入项目级 `.claude/settings.json`；启动 PTY 会话。

###### 1.1.3 闲聊入口
- **功能描述**：对使用 Claude Code 场景的补充--需要长时间持续工作的任务通过创建项目，临时问一些问题或流程不长的小任务通过闲聊。点击后弹出独立的 Claude 终端子窗口（非项目绑定）。
- **业务规则与约束**：闲聊产生的 token 归入全局统计的"未分类"项，不归属任何项目。
- **数据范围**：独立终端会话；token 统计。

###### 1.1.4 Plugins 面板 `[规划中·占位]`

##### 2.3 历史工作区

> 单个 Agent 的工作进程框，完整还原从指令输入、思考推理、落地执行到多体协作的全流程。是**问题根因回溯与 AI 幻觉事后排查**的核心载体。基于 JSONL 转录文件全量留存。

###### 2.3.1 主线 Agent 工作框

- **功能描述**：采用「时间中轴线 + 左右分层 + 分叉扩展」的可视化结构，共划分为四类核心事件。
- **业务规则与约束**（四类核心事件）：

  **① 人与 Agent 交互节点（时间中轴线）**——整条工作链路的主干节点，类比甘蔗分节与树干节点：
  - 以 Agent 的关键阶段性回复作为核心节点标记
  - 用户向 Agent 输入指令的入口
  - 通过这些节点可快速把握整条时间线的核心推进脉络，大颗粒度定位人工介入交互的关键位置

  **② Agent 思考层事件（中轴线左侧）**——对应 Agent 的内在认知与思维碰撞过程，所有事件沿时间中轴线左侧排布，skills / 工作流 / BTW / Insight 采用不同长度的插入线标识，便于快速识别与事后复盘：
  - 经验调用：Skills 调用、工作流调用（工作流目前仅通过 Hooks 实现，未来计划融入 n8n 工作流体系——agent 与 n8n 并非竞争关系，而是不同场景下的不同选择）
  - 思维碰撞节点：
    - **BTW**：基于 Prompt Cache 机制的临时试探性提问，用于用户对 Agent 工作过程产生疑问时快速交互，充分利用缓存降低 AI 成本；单次回答不进行 action，不增加原有对话上下文
    - **Insight**：Claude Code 原生机制，记录 Agent 解决问题过程中的关键突破点与核心洞见，相当于人工执行任务时最值得复盘沉淀的关键节点

  **③ Agent 行动层事件（中轴线右侧）**——对应 Agent 将思考落地执行的具体动作，所有事件沿时间中轴线右侧排布，且 tool / MCP / CLI 采用不同长度的插入线标识，可快速区分 Agent 调用的工具类别：
  - Agent 自带工具（Tools）
  - MCP 工具
  - CLI（识别为 Skill 包中带 cli 字的）

  **④ 多 Agent 分叉事件**——从任意主线交互节点均可触发创建多 Agent 分支，依托 Claude Code 原生的 branch / subagent / BTW 三个核心概念实现：
  - **Subagent**：由主 Agent 分配独立执行目标单一、路径清晰的专项任务（如资料检索、信息整理），主 Agent 仅接收最终结果，避免中间过程的无效信息占用上下文窗口。创建时支持配置：指派某个 subagent 的目标、设定新 subagent 的人设、subagent 的数量
  - **Branch**：用于路线试错、项目答疑或并行作业三类场景，从主线节点分叉出独立探索分支，完整记录探索过程与结果，不干扰主链路推进
  - **BTW**：本质是继承主线 Agent 上下文缓存的一次性轻量 Agent，对话结束即销毁，非常适合针对主线工作内容发起临时疑问与快速验证

- **Plan Tag（里程碑标记）**：当 Agent 完成某项制定的 Plan 里的某条（T 节点变 DONE），插入 Plan Tag（里程碑标记，如 `[M1-S1-T1]`），标记对应原子目标完成与阶段性成果。相关代码：`renderer/src/hooks/usePlanIndicator.ts`（检测 T 节点 DONE 生成里程碑）；里程碑持久化 `<claudeId>.milestones.jsonl`。
- **数据范围**：JSONL 转录文件（user/assistant 节点）+ 三通道事件（插入线）+ Plan 状态（Plan Tag）。

##### 2.4 Plan 区
- **功能描述**：长期项目的实现计划区，显示当前工作进展（已完成的、未完成的、现在在进行哪个子任务目标）。底层由「机制·Plan 管理」支撑（显示/追踪/状态，见 2.2 机制）。

###### 2.4.1 执行计划折叠区
- **功能描述**：实时工作区顶部已移至本区的折叠/展开 Plan 面板。折叠态显示当前 Plan 节点 badge；展开态显示 Plan 树（左列）+ 阅读窗格（右列）。
- **业务规则与约束**：折叠态高度紧凑，展开态有最大高度限制；历史进度拉动条移动时 Plan 指针同步更新。
- **数据范围**：项目 `plan/overall_plan.md`。

###### 2.4.2 Plan 三级管理与格式
- **业务规则与约束**：
  - **三级管理**（也可自定义更多级，默认三级）：母任务(M) - 子任务(S) - 原子任务(T)。
  - **制定多级 Plan 的原因**：可以尽可能确保 Agent 不会偏离方向，每个 Plan 要有检查标准，只有通过了才可以进行下一个 Plan 的工作，这样可以保证 Agent 的工作像搭积木一样--之前搭建的积木是坚实的，再往上搭的时候假如搭错了，只会影响最上层的，下面的依然可以复用。
  - **Plan 文件格式**（**重要：与全局/项目监控界面的 Plan 面板解析逻辑强绑定，格式不正确将导致无法识别**）：
    | 层级 | 必须写成 | 示例 |
    |------|---------|------|
    | M（母目标） | `## M<数字> - <标题> [状态]` | `## M1 - 项目脚手架与基础架构层 [ ]` |
    | S（子目标） | `### S<数字> - <标题> [状态]` | `### S1 - electron-vite 项目初始化 [x]` |
    | T（任务） | `- [<状态字符>] T<数字> - <标题>` | `- [!] T2 - 实现双向 invoke/handle 机制` |
    状态字符：`[ ]` 待办 / `[x]` 已完成 / `[!]` 进行中。
  - Plan 文件路径：相对于项目根目录，统一命名 `plan/`，主文件为 `overall_plan.md`。
  - 结合 Git 功能，可以实现更高效更安全的 Agent 工作模式。
- **数据范围**：项目 `plan/` 目录下的 Markdown 文件。

##### 2.5 Git 功能

#### 概念三：消息通知界面（独立通知窗口）

> 独立窗口形式的聚合通知中心，与实时监控面板解耦。窗口真正独立于主窗口（主窗口最小化不影响它），始终置顶，按"正在运行的项目"自动分割展示权限请求，另有"系统通知"分区展示 insight 报告等系统级通知。

- **功能描述**：独立的系统级通知窗口（非主窗口内的 tab/面板）。窗口内分两类内容：① 按"正在运行的项目"纵向分割为多个子区，每个子区顶部为项目名，下方独立滚动显示该项目的权限请求；② "系统通知"分区，展示 insight 报告完成等系统级通知。每个权限请求以 2 行紧凑形式展示：第一行为 Agent 框名称 + 调用名称 + 展开按钮 + 关闭按钮；第二行为审批交互（同意 / 同意+附加消息 / 拒绝 / 拒绝+附加消息）。点击展开按钮显示工具调用详情（与历史工作面板触发线一致的可视化：不同类型不同颜色）。Insight 通知以 2 行展示：标题 + 报告路径，点击"查看报告"在浏览器中打开。
- **业务规则与约束**：
  - **独立窗口**：不随主窗口最小化而隐藏；始终置顶（可配置，默认开）；默认尺寸为 400×400，最小尺寸同为 400×400，允许用户继续放大窗口。
  - **单例窗口**：全局唯一，不重复创建；已打开则聚焦。
  - **三个打开途径**：① 用户点击顶栏"打开独立通知窗口"按钮；② 收到权限请求时自动打开（可配置默认开，即使窗口已隐藏到托盘）；③ insight 报告完成时自动打开（窗口不存在则创建，已存在则聚焦）。
  - **关闭按钮**：隐藏到托盘（窗口存活，状态保留）；新权限请求来时恢复显示并抢焦点置顶。
  - **按项目分割**：分割区数量 = 正在运行的项目数（运行中判定：项目有至少 1 个 Agent Card 在实时工作区可见）。项目停止运行时，对应分割区及其通知全部移除。
  - **2 行布局**：① Agent框名称（主线程 / Agent(xxxxxx)，subagent 显示 agent_id）+ 调用名称 + 展开按钮 + 关闭按钮；② 4 交互 Yes/No（同意 / 同意+消息 / 拒绝 / 拒绝+消息，消息随输入框发送，逻辑同原审批卡片；底层为 TUI 按键序列：同意=回车；拒绝=方向键下×2+回车；附加=Tab+文字+回车）。
  - **展开详情**：默认只显示调用名称，点击展开按钮显示工具调用完整详情（字段、命令、文件路径等），可视化与历史工作面板触发线一致（不同类型不同颜色）。
  - **关闭操作**：每个权限请求条目支持点击"关闭"按钮，关闭后通知从列表消失，但 Agent 继续等待（不会收到审批响应）。角标计数同步更新。
  - **通知范围**：权限请求 + insight 报告完成通知（info/warning/error 类通知不在此窗口展示）。
  - **可配置性**：窗口"始终置顶"和"权限请求时自动打开"两个行为可在全局设置·通知分区开关控制（默认开）。
  - **系统通知**：桌面通知 + 任务栏角标（跨平台）保持不变。
- **数据范围**：权限请求事件（含工具调用详情）；正在运行的项目（基于实时工作区可见的 Agent Card 派生）。

### 2.2 机制详细说明
> 机制是为了实现概念而在底层运行的功能，用户没有直接接触，但保障了概念功能的正常运行。一个机制可被多个概念复用。

#### 机制一：多 Agent 点对点管理
- **功能描述**：一个统一界面对多 Agent 的点对点管理，是整个软件的底层核心机制。贯穿全局监控与项目监控。
- **业务规则与约束**：
  - **单个命令终端的识别 / 创建 / 写入 / 读取 / 生命控制**：
    - 识别：通过 Session ID 唯一标识每个 Agent 会话
    - 创建：四种启动模式（新建 / 恢复 / 裸启动 / 命令启动）
    - 写入：通过 stdin 管道向会话注入指令（含消息队列注入、权限审批注入）
    - 读取：三通道读取（Hooks 主事件流 / statusLine 实时 / JSONL 历史回看）。**读取后衍生存储**：除原始读取数据外，额外持久化 5 类衍生数据用于重启回放与跨会话关联--① `<claudeId>.insertions.jsonl`（十类插入线完整 record + patch）；② `<claudeId>/subagents/agent-*.insertions.jsonl`（subagent 插入线）；③ `<claudeId>.milestones.jsonl`（Plan Tag/里程碑，含 DOM 高度快照）；④ `<claudeId>.git-marks.jsonl`（git 快照对应 commit）；⑤ `<claudeId>.meta.json`（session startedAt）。此外 **btw 问答配对**复用 insertions.jsonl 持久化（subagentHandler 回填）。
    - 生命控制：心跳检测（定期检测进程存活）、超时关闭（长时间无交互自动关闭）
  - **多个命令终端的统一管理**：点击回到对话、实时工作区和历史工作区的对应机制、同项目不同 Agent 之间的切换（快速切换与精细化切换）`[已实现]`（via useGlobalKeyNav，见 2.3.5）。
  - **对话生命周期管理**：
    - 所有 Session 必须通过仪表盘启动（才能完整管控）
    - 终端子窗口可关闭不终止 PTY 进程（仅取消渲染，重新打开可恢复）
    - 手动启动的 Claude 进程：仪表盘可扫描 JSONL 转录文件只读显示，但**无法注入指令**
    - 长时间无新交互自动关闭（资源释放）+ 发出提醒通知
- **相关代码**：`main/lib/pty/PtyManager.ts`（start/resume/stop/write/resize）、`main/index.ts`（衍生存储 IPC）、`renderer/src/business/`（各 handler）、`renderer/src/capabilities/ptyBindings.ts`。
- **数据范围**：PTY ↔ Claude Session 双向绑定表；Session 元数据；JSONL 转录；衍生持久化文件。

#### 机制二：信息分类系统
- **功能描述**：对 Agent 工作过程产生的信息进行分类与分层展示，按场景控制信息密度（渐进式披露），是"可视化驾驭"的底层支撑。对应设计哲学中"信息的输出（展示）"。
- **业务规则与约束**：
  - **四类核心事件分类**：人与 Agent 交互节点（时间中轴）/ Agent 思考层事件（左侧）/ Agent 行动层事件（右侧）/ 多 Agent 分叉事件。
  - **十类插入线分层**：思考层（skill/workflow/insight/subagent/branch/btw/AskUserQuestion）与行动层（tool/mcp/cli）采用不同方向、长度、颜色标识，快速区分信息类别。
  - **信息密度可调节**：每个场景下只显示当前所需的必要信息，详情放入可展开区域，支持不同颗粒度查看。
- **相关代码**：`renderer/src/features/project-monitor/ProcessTimeline.tsx`、`LineInsertionItem.tsx`（十类插入线）、`canvas/ProcessLineCanvas.tsx`。
- **数据范围**：来自三通道事件 + JSONL 转录。

#### 机制三：Token 捕获
- **功能描述**：从 Claude Code 获取每个对话的 Token 用量，是成本统计的数据来源。对应设计哲学中"信息的产生（捕获）"。
- **业务规则与约束**：
  - **唯一计费源**：JSONL 转录文件的 `message.usage` 字段（提取 input/output/cache_creation/cache_read tokens + model）。
  - **两源捕获**：① 实时增量（tail 监听 JSONL 追加）；② 历史全量扫描（读整个 transcript 累加，与实时值取大去重）。
  - **非计费源**：statusLine 的 context_window 是上下文窗口占用率，非累计消耗 token；`/status` 命令未用作计费源。
  - **subagent token 未捕获**。
- **相关代码**：`main/lib/jsonl/JsonlParser.ts:87-99`、`JsonlWatcher.ts:296`、`main/index.ts:2601-2648`（TOKEN_SCAN）、`renderer/src/business/jsonlHandler.ts`、`capabilities/tokenCapability.ts`。

#### 机制四：Token 统计
- **功能描述**：将捕获的 Token 按对话/项目/全局聚合计算，并换算为费用。
- **业务规则与约束**：
  - **内存聚合**：全部在 Jotai 派生 atom 中计算，无持久化（重启归零，靠重新扫描 JSONL 恢复）。
  - **费用公式**：`(input+cacheCreation+cacheRead) / 1M × input单价 + output / 1M × output单价`（cache token 按 input 单价计费）。
  - **当前限制 `[部分实现]`**：① 聚合范围仅当前内存活跃 session，未加载历史 session 不计入"累计"；② 时间分桶用 session 开始时间非 token 消耗时间；③ subagent token 未计入；④ 不区分 provider。
- **相关代码**：`renderer/src/atoms/stats.atom.ts`、`capabilities/tokenCapability.ts`、`features/settings/sections/TokenCostSection.tsx`。

#### 机制五：系统通知推送
- **功能描述**：将关键事件（权限请求、insight 报告完成）以桌面通知 + 任务栏角标 + 独立通知窗口形式推送，避免用户错过 Agent 阻塞或重要报告。对应"信息的输出"与"通知统一"。
- **业务规则与约束**：
  - **桌面通知 + 跨平台任务栏角标**（Windows setOverlayIcon / macOS dock.setBadge / Linux setBadgeCount）。
  - **当前硬编码绑定权限请求**：角标计数语义为"待处理权限请求数"（PermissionRequest 触发 notify+increment，审批后 decrement），非通用通知。
  - **关闭操作也会更新角标**：用户点击"关闭"按钮后，角标计数 -1（与审批操作相同），但不发送任何按键到 PTY。

  - **应用内通知**（独立通知窗口）与桌面通知是两套，仅通过权限请求队列间接关联。权限请求来时自动打开独立通知窗口（若已隐藏则恢复+抢焦点；该自动打开行为可在全局设置·通知分区开关，默认开）。Insight 报告完成时同样自动打开通知窗口。

- **相关代码**：`main/lib/notification/NotificationService.ts`、`main/index.ts`（PermissionRequest 触发 / INSIGHT_REPORT_READY 广播）、`renderer/src/atoms/notification.atom.ts`。

#### 机制八：增量原子写入配置
- **功能描述**：只管理特定字段，不是重写文件全部内容，不会影响原有的其他部分内容。解决"修改配置项担心覆盖其他内容"的痛点。
- **业务规则与约束**：
  - **写临时文件 + 原子重命名**：先把新内容完整写到一个临时文件，全部写完后用"原子重命名"操作把临时文件替换原文件。这样要么完整替换成功、要么完全不变，绝不会出现写到一半的损坏文件（即使写入中途断电/崩溃，原文件也还在）。
  - **字段级合并**：只改你要改的字段，其他字段原样保留。举例：要改 settings.json 里的 model 字段，先读出整个文件，只把 model 改成新值，其他字段（permissions、env 等）都不动，再整体写回。
  - 适用所有配置：项目记录单、全局配置、CC settings.json、`~/.claude.json`、定时任务、cc-connect 配置。
- **相关代码**：`main/lib/config/SettingsManager.ts`、`ClaudeJsonManager.ts`、`DriverConfigStore.ts`、`main/lib/projects/ProjectStore.ts`（统一 write-tmp + rename 模式）。

## 3. 交互流程和细节

### 3.1 首次打开·初始化 SOP

**前置条件**：
- 首次启动软件，项目记录单为空

**主流程**：
1. 软件检测依赖（Node.js ≥18 / npm / Git / Claude Code CLI）→ 若缺失则引导安装（支持 Claude CLI 自动安装）
2. 通过初始化 SOP 弹窗（Modal）引导用户选择"存在所有项目的文件夹"。相关代码：`renderer/src/features/global-monitor/InitSopModal.tsx`。
3. 递归扫描全部含 `CLAUDE.md` 的文件夹（排除 node_modules 等，最大深度 6）→ 写入项目记录单并标记
4. 向用户发送项目认领清单
5. 用户对每个项目选择认领状态：`1` = 确认是项目 / `0` = 不是项目 / `-1` = 稍后决定（默认）
6. 后续每次打开：加载标记为 1 的项目；检测到新文件夹且未标记 → 发送确认清单

**反馈规则**：
- 依赖缺失 → 弹窗提示并引导
- 扫描完成 → 显示认领清单
- 认领完成 → 进入全局监控主界面

### 3.2 新建项目·创建项目向导

**前置条件**：
- 用户在全局监控画板点击"+ 新建项目"

**主流程**：
1. **Step 1·项目设置**：输入项目名称（加入项目记录单）→ 以按钮形式选择可调用的 Skills/MCP/Tools/Agent/Plugins → `[待开发]`（Step 1 实际仅含 名称/路径/描述/权限；"选择可调用 Skills/MCP/Tools/Agent/Plugins"与"系统提示词配置界面"均待开发，详见 1.1.2）选择权限（6 选 1，悬停显示说明）→ 可选展开系统提示词配置界面（填入模板变量 `{{PROJECT_NAME}}` / `{{PROJECT_DESCRIPTION}}` / `{{ENVIRONMENT}}` 等）
2. **Step 2·放入资产**：提示放入调研报告、demo 代码等资料到资料文件夹 → 点击"已放好"
3. **Step 3·制定计划**：自动跳转到项目监控页面 → 自动在项目文件夹打开新 Claude Session（权限 acceptEdits）→ 弹出 Claude 终端子窗口 → 自动发送"创建计划"指令（通过 PTY stdin 注入）

**反馈规则**：
- Session 启动成功 → 终端子窗口出现，开始创建计划
- Session 启动失败 → 静默处理 `[待确认·是否需要更明确反馈]`

### 3.3 项目监控·实时驾驭 Agent

**前置条件**：
- 已有项目，且至少一个 Agent 在工作

**主流程**：
1. 用户在项目监控界面实时工作区看到当前活跃 Agent Block（含状态、token、运行时长、当前工具/经验快照）
2. 用户在 Agent 专属消息输入行随时输入消息（不等待 Claude 完成）→ 消息入队
3. Claude 完成当前响应（Stop Hook 触发）→ 自动弹出队列最老消息注入会话
4. 若 Agent 发出权限请求 -> 独立通知窗口自动打开（置顶）并在对应项目分割区显示 -> 用户同意/拒绝（可附加信息）-> 发送 TUI 按键序列（同意=回车；拒绝=方向键下×2+回车；附加=Tab+文字+回车）
5. 用户点击打断按钮 → 发送 Ctrl+C 中断当前对话
6. 用户通过上下文面板查看该 Agent 当前上下文 → 判断输出是否可能为幻觉

**反馈规则**：
- 工具调用 → 工具大框实时刷新 + 工作状态词更新
- 权限请求 → 独立通知窗口自动打开（置顶），多请求按项目分割区 FIFO 堆叠
- 消息入队 → 队列指示
- 上下文变化 → 上下文面板实时更新

### 3.4 历史工作区·节点操作

**前置条件**：
- 项目有历史对话记录

**主流程**：
1. 用户在历史工作区浏览主线 Agent 工作框的时间线
2. 用户操作历史进度拉动条快速跳转到某个用户输入区间
3. 用户点击某个 Claude 回复节点 → 弹出 4 选项菜单
4. 根据选择：
   - **Git 当前版本** → 执行快照 → 节点标记特定颜色
   - **回退到当前版本**（仅已 Git 节点）→ 执行版本回滚
   - **删除**（仅用户输入节点）→ 删除区间 log（含 Git commit 则 rebase）
   - **创建多 Agent**（仅最后节点）→ 展开三子选项 → 选择 branch/指派 Agent/btw

**反馈规则**：
- 未 Git 节点"回退"灰色不可交互
- Claude 节点"删除"灰色不可交互
- 非最后节点"创建多 Agent"灰色不可交互
- 操作成功 → 节点视觉状态更新

- 已有足够使用历史

**主流程**：
1. 用户在全局监控功能入口点击"灵魂交流"
2. 选择"/insight 使用洞见"
3. 调用 `/insights`（复数）→ 等待生成
4. 报告完成 → 自动打开独立通知窗口，在"系统通知"分区推送通知（含 `file://` 报告路径）
5. 用户点击通知 → 在 WebView 中打开报告

### 3.10 维护者·版本发布

**前置条件**：
- 当前位于 `develop`，本地没有未提交修改，且已与测试仓库的 `develop` 同步
- 测试仓库与正式发布仓库均已配置且具备推送权限
- 自动构建与 Release 发布所需的认证和工作流可用

**主流程**：
1. 维护者调用版本发布入口并传入版本号（如 `1.0.1`）
2. 系统执行只读前置检查并反馈检查结果；任何条件不满足时停止，不修改项目
3. 第一个修改步骤将应用版本更新为传入版本，并在确认写入成功后输出完成日志
4. 系统按顺序创建该版本的 release 分支并提交版本变更 → 推送到测试仓库 → 推送到正式仓库 main；每一步成功后输出完成日志
5. 系统创建并推送带 `v` 前缀的版本 tag，等待对应自动构建及安装包发布完成
6. 系统确认该版本 Release 成功后输出完成日志，再切回 `develop` 并清理本地及测试仓库的 release 分支
7. 全部步骤成功后输出版本号、Release 结果和清理结果汇总

**反馈规则**：
- 每一步均输出“开始 / 完成”日志；失败时输出“失败”、失败命令或检查项及建议处理方式
- 只有当前步骤明确成功，才进入下一步
- 版本号非法、分支或 tag 已存在、工作区不干净、远程不同步、权限不足、构建失败或 Release 未生成时，立即停止
- 失败后不自动回滚已经完成的远程操作，也不自动删除 release 分支，避免丢失可诊断现场

## 4. 边界情况与异常处理

### 4.1 边界情况

1. **Plan 文件损坏 / 格式不符**：直接不显示该条目，不报错崩溃（保证健壮性）。
2. **手动启动的 Claude 进程**：仪表盘可扫描 JSONL 转录文件只读显示，但无法注入指令（管控边界）。
3. **终端子窗口关闭**：不终止 PTY 进程，仅取消渲染输出；重新打开窗口可恢复 PTY 输出流接入。
4. **终端子窗口长时间无交互**：自动关闭（资源释放）+ 发出提醒通知。
5. **进程消失**：心跳检测发现进程消失 → 标记为"已中断"状态。
6. **多个 Agent 同时权限请求**：独立通知窗口内按项目分割区 FIFO 堆叠，无超时。
7. **Subagent 内部工具调用可见性**：归属于 Subagent 的 agent_id，主 Session Hooks 行为看不到，需在子 Agent Hook 事件中区分。
8. **删除节点区间含 Git commit**：同步执行非交互式 rebase 删除对应历史，禁止交互式 rebase。
9. **已 Git 节点也可删除**：不保护，删除后 Git 历史同步清除。
10. **未 Git 节点回退限制**："回退到当前版本"显示灰色不可交互。
11. **非最后节点创建多 Agent 限制**："创建多 Agent"显示灰色不可交互。
12. **JSONL 重复读取**：采用 tail 追加模式（文件监听 + 定时轮询兜底），避免重复处理已读内容。
13. **Plan 状态同步兜底**：独立线程定期全量读取所有 Plan 文件，修正 Hooks 漏报差异。
14. **文件监听静默停止**（部分平台长时间运行）：配置定时轮询兜底。
15. **Git 远程未配置 / 权限不足**：弹出子窗口说明操作步骤。
16. **CLI 识别规则**：在用户的 `.claude/skills` 下识别所有带 cli 字的 skill 为 CLI，不放入 skill 类。
17. **本月 Token 不区分 provider**：当前只支持不区分 provider 的统计，按 provider 区分为未来优化方向。
18. **Hook 注册合并**：首次启动自动注入 Hook 配置，合并到已有 hooks 配置（不覆盖用户其他 hooks）。
19. **版本发布幂等限制**：目标 release 分支、版本 tag 或对应 Release 已存在时，不覆盖、不复用，前置检查失败并停止。

### 4.2 异常处理

| 异常场景 | 处理策略 |
|---------|---------|
| 依赖缺失（Node/npm/Git/Claude CLI） | 弹窗提示 + 引导安装（Claude CLI 支持自动安装） |
| Plan 格式错误 | 跳过该条目，不崩溃 |
| 进程异常消失 | 心跳检测标记"已中断"，用户可"回到对话"恢复 |
| Git 操作失败 | 弹出子窗口说明操作步骤 |
| 版本发布步骤失败 | 立即停止后续步骤，输出失败步骤和恢复提示，并保留 release 分支与已完成操作现场 |
| cc-connect 未安装 | 检测并提示，提供 CLI 命令供手动安装 |
| /branch 检测异常 | Ctrl+C 强制退出 → SessionEnd → `claude -r` 恢复链路 |
| Session 启动失败 | 静默处理 `[待确认·是否需更明确反馈]` |
| 端口被占用 | `[待确认·具体处理方式]` |

### 4.3 降级策略

1. **Hooks 失效降级**：当 Hooks 主链路失效时，兜底链路定期全量读取 Plan 与 JSONL，保证数据不丢失。
2. **statusLine 首次调用前降级**：首次 API 调用前 context_window.current_usage 为 null，统计显示占位。
3. **手动启动进程降级**：只读显示，不提供注入能力（明确的能力边界，而非报错）。
4. **未开放功能降级**：Auto-Dream 等未开放功能点击显示友好提示，不写入配置，待官方开放后替换为真实功能。

## 5. 状态标识规范

全局统一的 6 种状态标识，贯穿项目卡片、Agent Block、Plan 节点等所有可视化元素：

| 可能暂停 | M 任务未完成且长时间无变动 | 暂停指示 |
| 出问题 | 发生异常 | 异常指示 |
