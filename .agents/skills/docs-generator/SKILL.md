# 项目文档生成器 (Docs Generator)

## 概述

本技能按照 **MRD → PRD → Architecture → TDD** 的瀑布流程，从产品方向到技术实现，逐层生成完整项目文档。核心特点：


## 要求：
1. 按 MRD → PRD → Architecture → TDD 瀑布流程逐步生成项目文档，每阶段完成后暂停等待用户确认后再进行下一个文档的编写
2. 所有生成的文档放置在项目的 `docs/` 目录下。
3. 当需求不确定的时候或没有争取用户同意的时候一定要暂停询问用户需求细节想法

---

## 核心流程和文档定位

```
MRD (方向) → PRD (产品) → Architecture (系统) → TDD (实现)
```

| 文档 | 核心问题 | 关注点 | 不讨论 |
|------|---------|--------|--------|
| **MRD** | 为什么值得做？（Why） | 产品机会 | 具体功能 |
| **PRD** | 产品应该是什么？（What） | 产品能力 | 技术实现 |
| **Architecture Design** | 系统应该如何组织？（System） | 系统架构 | 接口、数据库细节 |
| **TDD** | 每个模块如何实现？（Implementation） | 技术实现 | 产品价值 |

---

## 执行规则（必须严格遵守）

1. **严格瀑布顺序**：MRD → PRD → Architecture → TDD，不可跳过、不可并行
2. **每阶段强制暂停确认**：完成一个文档并写入 `docs/` 后，必须停下来向用户呈现文档摘要，明确询问「文档已生成，是否确认通过？」，等待用户明确表示确认后，才进入下一阶段
3. **内容来源真实**：文档内容必须从代码上下文（代码注释、README、已有设计文档、配置文件）中提取和聚合。若某些信息在代码中找不到，必须暂停询问用户，**禁止凭空编造**
4. **输出位置统一**：所有文档写入 `docs/` 目录，文件名分别为 `MRD.md`、`PRD.md`、`Architecture.md`、`TDD.md`
5. **基于上层展开**：PRD 必须基于已确认的 MRD 展开；Architecture 必须基于已确认的 PRD 展开；TDD 必须基于已确认的 Architecture 展开

## 各文档模板

详细的各文档章节模板参见 `references/` 目录：

| 模板文件 | 用途 |
|---------|------|
| `references/mrd-template.md` | MRD 章节结构与内容要求 |
| `references/prd-template.md` | PRD 章节结构与内容要求 |
| `references/architecture-template.md` | Architecture Design 章节结构与内容要求 |
| `references/tdd-template.md` | TDD 章节结构与内容要求 |

## 工作流

### Phase 1: MRD — 市场/用户需求文档

**目标**：回答「为什么值得做？」，聚焦产品机会，不讨论具体功能。

1. **调研分析**：根据用户的指示和输入，理解当前产品的目标用户和使用场景，如果有不确定或者想质疑的点随时提出。不用阅读代码类文件
2. **按模板生成**：参照 `references/mrd-template.md` 的章节结构，生成 `docs/MRD.md`
3. **暂停确认**：向用户呈现 MRD 文档的核心摘要（Persona、Problem Statement、产品机会），明确询问确认
4. 用户确认通过后，进入 Phase 2；若用户提出修改意见，修改后再次确认

### Phase 2: PRD — 产品需求文档

**目标**：回答「产品应该是什么？」，聚焦产品能力，不讨论技术实现。

1. **基于 MRD 展开**：根据用户的指示和输入以及已确认的 MRD 为输入，将痛点拆解为功能需求
2. **按模板生成**：参照 `references/prd-template.md` 的章节结构，生成 `docs/PRD.md`
3. **暂停确认**：向用户呈现 PRD 的核心摘要（需求→功能映射、概念与机制列表、交互流程概要），明确询问确认
4. 用户确认通过后，进入 Phase 3

### Phase 3: Architecture Design — 架构设计

**目标**：回答「系统应该如何组织？」，聚焦系统架构，不讨论接口和数据库细节。

1. **基于 PRD 展开**：根据用户的指示和已确认的 PRD 为输入，设计或总结现有的系统架构，若有代码，说明用户先开发了代码，阅读代码之后再写，且阅读时可以以代码文件夹的结构和PRD为脉络一点点的仔细阅读对应代码
2. **按模板生成**：参照 `references/architecture-template.md` 的章节结构，生成 `docs/Architecture.md`
3. **暂停确认**：向用户呈现架构核心摘要（Overall Architecture、Module Boundary、Technology Selection），明确询问确认
4. 用户确认通过后，进入 Phase 4

### Phase 4: TDD — 技术设计文档

**目标**：回答「每个模块如何实现？」，聚焦技术实现细节。

1. **基于 Architecture 展开**：根据用户的指示和已确认的 Architecture 为输入，设计或总结现有的TDD，若有代码，说明用户先开发了代码，阅读代码之后再写，且阅读时可以以代码文件夹的结构和Architecture Design为脉络一点点的仔细阅读对应代码细化每个模块的实现
2. **按模板生成**：参照 `references/tdd-template.md` 的章节结构，生成 `docs/TDD.md`
3. **暂停确认**：向用户呈现 TDD 核心摘要（模块列表、API 概览、Data Model），明确询问确认
4. 全部确认通过，文档生成流程完成

---

## 注意事项

- 每一层的文档只包含该层应有的内容，严格遵循「不讨论」列的约束
- 若项目已有部分文档，应在已有文档基础上补充和更新，且需要争求用户统一，而非直接覆盖
- 文档中的不确定信息（如无法从代码中确认的参数、配置值、业务规则），必须标注 `[待确认]` 并向用户询问
- 生成文档后向用户呈现摘要时，重点突出该层的核心决策和关键内容，而非逐字复述文档
