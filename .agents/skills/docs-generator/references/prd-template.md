# PRD 模板 (Product Requirements Document)

## 核心定位

PRD 回答「**产品应该是什么？（What）**」，聚焦**产品能力**，不讨论技术实现。

---

## 章节结构

### 1. 痛点 → 功能映射（需求拆解）

将 MRD 中的痛点拆解为多个子功能。支持**多级结构**，直到拆解到原子功能/能力。

- **概念（Concept）**：用户直接看到和用到的功能。上一级是对下一级的聚类分类。概念之下可以有子概念，形成多级结构。
- **机制（Mechanism）**：为了实现概念而在底层运行的功能，用户没有直接接触，但保障了概念功能的正常运行。

此章节只列概念和机制的**名称**及**需求/痛点和（概念&机制）的对应关系**，一个需求可以由多个概念和机制满足，一个概念和机制也可以用在不同的需求上，按照实际情况构建对应关系，不过不需要描述细节，细节描述在下一章节展开。

示例格式：

```
[需求一]--->[概念一]----->[下一级概念]
        |           |--->[下一级概念]
        |-->[概念二]---->[下一级概念]
        |            |--->[下一级概念]
        |            |--->[下一级概念]
        |-->[机制1]
        |-->[机制2]
        |-->[机制3]
[需求二]--->[概念二]--->[下一级概念]
        |-->[机制1]
```



### 2. 概念和机制详细说明

- **概念（Concept）**：用户直接看到和用到的功能。上一级是对下一级的聚类分类。概念之下可以有子概念，形成多级结构。支持**多级结构**，直到拆解到原子功能/能力
- **机制（Mechanism）**：为了实现概念而在底层运行的功能，用户没有直接接触，但保障了概念功能的正常运行。
概念中会有机制，机制中会有概念，概念是多级的，在不同颗粒度下服务于某一个作用的标签，所以标签内部可以继续分出一些子标签，可以不断递归；而机制描述的是规律/规则（不管是人为制定还是自然制定，比如供需关系，这个就是一个规律）

以概念为脉络搭骨架多级层级结构，说明功能细节，不涉及技术实现细节。

对每个概念和机制，描述：

- **功能描述**：这个功能是什么
- **业务规则与约束**：所有确定性规则，如：
  - "订单完成后 7 天内可评价，超时关闭入口"
  - "评价最多上传 9 张图"
  - "敏感词内容进入人工审核"
  - ...等所有确定性业务规则
- **数据范围**：涉及哪些数据，数据从哪里来

包含2类内容，一个是骨架（只包含名字，不包含细节，目的是从全局视角展示架构以及各个之间的关系），一个是根据骨架脉络列出的细节，一个机制可以用于多个概念中按照实际情况对应关系


示例格式：

```
> 骨架
[概念一]--->[下一级概念]----->[再下一级概念]...
        |              |--->[再下一级概念]...
        |              |--->[机制一]
        |---------------------------------------------------------
        |-->[下一级概念]                                          |
        |-->[下一级概念]---->[再下一级概念] ...                    |---->机制二
[概念二]--->[下一级概念]----->[再下一级概念] ...                    |
        |              |--->[再下一级概念]-------------------------
        |
        |-->[下一级概念]---->[再下一级概念]...

[概念三]--->[机制]----->[再下一级概念]...
        |         |--->[再下一级机制]...
        |
        |-->[下一级概念]---->[再下一级概念]...
> 细节
## 概念1
### 下一级概念
#### 再下一级概念
#### 机制2
##### 再下一级概念（按实际情况决定拆几层，反正最后是原子概念）
### 机制1

## 概念2

## 机制1
...
```

### 3. 交互流程和细节

对每个概念，描述用户实现核心功能的完整路径：

- **核心动作的完整路径**：用户从入口到完成的全流程步骤
- **前置条件**：触发某个功能需要满足什么条件
- **交互反馈规则**：每个操作后系统给出什么反馈（成功/失败/进行中/异常）

示例格式：

```markdown
### 概念 [名称] 的交互流程

**前置条件**：
- [条件1]
- [条件2]

*主流程**：
1. 用户[动作1] → 系统[反馈1] →if[条件]→动作→反馈·
                           →elseif[条件]→反馈
2. 用户[动作2] → 系统[反馈2]→ ..
3. ...
```

## 约束

- 不讨论技术实现（留到 Architecture 和 TDD）
- 不讨论「用什么技术栈」「数据库怎么设计」等技术选型问题
- 只从功能层面描述产品应该提供什么能力
- 不确定的业务规则标注 `[待确认]`
