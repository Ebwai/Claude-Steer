# SOP：有代码项目 → Architecture + TDD 文档体系

## 前置条件
- 项目有**实际代码库**
- 有 PRD（`docs/PRD.md`，已由 docs-generator 生成或已有）
- block_sync 已部署，hideRule=selective + keepContent 白名单

## 核心原则（代码优先，PRD 校正）

1. **代码是真相源**：文档描述的是代码实际的行为、结构和接口，不是 PRD 的理想设计
2. **PRD 是参照**：PRD 定义"应该有什么"，用于对照发现差异
3. **差异先记录，最后汇报**：过程中不打断，全部记录到差异清单，最后统一向用户汇报
4. **块同步 = 自动化上下级传递**：每填完一级，sync 自动级联，不需要手动更新上层汇总

## 流程（7 步）

### Step 1: 读 PRD，提取概念清单（约 10% 时间）

1. 通读 `docs/PRD.md`
2. 提取所有**概念**（concept）和**机制**（mechanism）的名称
3. 记录每个概念的：
   - PRD 中的名称
   - 所在 PRD 章节
   - PRD 中的职责描述（一句话）
   - 与哪些概念有关联
4. 创建"PRD 概念清单表"（作为后续对照的基准）

输出：PRD 概念清单。

### Step 2: 探索代码库，扫描实际结构（约 30% 时间）

1. **扫描文件夹树**（到叶子级，跳过 node_modules/build/out/dist 等非代码目录）
2. **识别模块**：每个含代码文件的文件夹 = 潜在模块
3. **提取每个模块的实际信息**（直接从代码中获取，不靠猜测）：
   - 模块名（文件夹名）
   - 职责（从代码注释、README、文件名、类名/函数名推断）
   - 内部文件列表
   - 对外的 API 签名（export/interface/public methods）
   - 依赖的其他模块（import/require 语句）
   - 数据模型（type/interface/class 定义）
4. **并行效率**：可以并行读多个叶子模块的代码（不同文件，不冲突）

输出：代码实际模块清单 + 每个模块的原始信息。

### Step 3: 对照 PRD vs 代码，找出差异（约 10% 时间）

将 Step 1 的 PRD 概念清单与 Step 2 的代码实际模块清单逐项对照，记录所有差异。**只记录，不修改任何文件**。

#### 差异类型及处理

| 类型 | 判定 | 处理 |
|------|------|------|
| **代码有、PRD 无** | 代码中有该模块/功能，但 PRD 未提及 | 记录到差异清单，按代码实际填写块内容。标注 `[PRD未提及]` |
| **PRD 有、代码无** | PRD 描述了该概念，但代码中找不到对应实现 | 记录到差异清单。代码中不建对应块文件。标注 `[规划中]` 或 `[已移除?]` |
| **名称不一致** | 同一概念，PRD 和代码用不同名称 | 记录到差异清单。**以代码实际名称为准**填块文件。在差异清单中列出对照 |
| **职责不匹配** | PRD 描述的职责与代码实际行为不符 | 记录到差异清单。**以代码实际行为为准**填块内容。标注差异 |
| **模块边界不同** | PRD 的模块划分层级与代码实际文件夹结构不同 | 记录到差异清单。**以代码实际结构为准**。标注差异 |
| **已废弃/重构** | 代码中有废弃代码或明显的重构痕迹 | 记录到差异清单。按**当前有效代码**填写。废弃代码不纳入 |

#### 差异清单格式（Step 7 汇报用）

```markdown
## PRD vs 代码 差异清单

### 代码有、PRD 未提及
| 模块名 | 代码路径 | 推断职责 | 建议 |
|--------|---------|---------|------|
| ... | ... | ... | 纳入 PRD / 待确认 |

### PRD 有、代码中未找到
| 概念名 | PRD 章节 | 可能原因 | 建议 |
|--------|---------|---------|------|
| ... | ... | 规划中 / 已废弃 / 名称不同 | 待确认 |

### 名称不一致
| PRD 名 | 代码名 | 采用 | 理由 |
|--------|--------|------|------|
| ... | ... | 代码名 | 代码实际使用 |

### 职责/边界差异
| 模块 | PRD 描述 | 代码实际情况 | 建议 |
|------|---------|------------|------|
| ... | ... | ... | 以代码为准 / 需改代码 |
```

### Step 4: 确定模块级边界

结合三项输入综合判定：
1. 代码实际文件夹结构（Step 2）
2. PRD 的概念层级（Step 1）
3. 架构.md 中的模块定义（如有）

**判定原则**：
- Architecture 到"代码中包含独立子模块的文件夹"为止
- TDD 从"包含可直接描述实现细节的代码文件"的文件夹开始
- 两体系在模块级交汇：同一文件夹既有 Architecture 块（描述定位/组成/联动）又有 TDD 内容（描述 API/数据模型/流程）

**用户确认**：AI 判定后必须列出模块级清单，请用户确认。

### Step 5: 建文档骨架 + 配 registry

1. 创建所有块文件（`.claude/rules/architecture/` 镜像目录）
   - `paths` frontmatter 指向**代码实际文件夹**的相对路径
   - 内容先留占位（`<!-- TODO: 待填充 -->`）
2. 创建顶层汇总（`docs/architecture.md` + `docs/TDD.md`）
3. 配 `registry.json`（递归 aggregate + selective hideRule）
4. 验证：`python -m app.registry` 确认加载无误

### Step 6: 从叶子开始，读代码填内容（约 40% 时间）

按依赖顺序（无依赖的叶子先填），逐模块填写：

#### TDD 叶子模块

对每个叶子模块（最底层含代码文件夹）：
1. 打开对应的代码文件（按 paths frontmatter 指向的目录）
2. 按 `tdd-template.md` 写块内容：
   - **导航类**（内容保留，级联传递）：
     - `### 模块架构图`：mermaid 画该模块内部文件/组件的依赖关系（从 import 提取）
     - `### 模块概览`：从代码注释/文件名/类名提取职责、输入、输出
     - `### API 概览`：从 export/interface 提取关键签名（不写参数默认值细节）
   - **细节类**（只留标题）：数据模型、关键流程、状态机、异常处理、监控与测试
3. **每填完一个叶子** → sync 自动级联更新中层汇总

#### Architecture 中层

叶子全部就绪后，逐级向上：
1. `### 内部组成`：列出子模块名 + 各自一句话作用（从下级块文件的内容提取）
2. `### 依赖与联动`：描述本级内部子模块的依赖关系和通信方式
3. `### 架构图`：mermaid 画子模块依赖拓扑
4. `### 定位与职责`

#### 关键效率点

**块同步让填内容 100% 聚焦在"当前这一级的块文件"上。** 你不需要：
- ❌ 手动更新上层汇总
- ❌ 复制粘贴内容到上级
- ❌ 担心上下级不一致

你只管填好当前级的块文件，sync 自动把导航内容级联传递到所有上层。

### Step 7: 手写顶层 + 汇报差异

1. **手写 `docs/architecture.md` 的 `## 整体架构`**：设计理念、关键决策、全局技术选型、顶层模块依赖图
2. **手写 `docs/TDD.md` 的 `## 概述`**：设计原则、测试策略、统一规范
3. **整理 PRD 差异清单**：汇总 Step 3 记录的所有差异，用 Step 3 的模板格式化
4. **向用户汇报**：呈现差异清单 + 建议（哪些需要更新 PRD、哪些需要改代码、哪些需要确认）

## 效率最大化总结

| 阶段 | 时间占比 | 关键动作 |
|------|---------|---------|
| Step 1 读 PRD | 10% | 快速扫，建概念地图 |
| Step 2 扫代码 | 30% | 并行读叶子模块，提取实际结构 |
| Step 3 diff 对照 | 10% | 快速比对，记录差异 |
| Step 4 定模块级 | 5% | AI 判定 → 用户确认 |
| Step 5 建骨架 | 5% | 一次性创建 + 配 registry |
| Step 6 填内容 | 40% | 从下往上，每填一级 sync 自动级联 |
| Step 7 汇报 | 5% | 差异清单 + 顶层手写 |
