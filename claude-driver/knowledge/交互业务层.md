# 交互业务层设计文档

> **定位**：业务层负责编排能力层，定义每个用户交互或系统事件的完整时序。
> 业务层了解"这个交互需要哪些能力、按什么顺序、用什么条件分支"，但不自己实现这些能力。
> 能力层的 API 是稳定的契约，业务层的时序设计可以随需求迭代而调整。

---

## 设计原则

1. **时序在业务层**：竞态处理、顺序依赖、条件分支全部写在业务层，能力层不感知顺序
2. **能力层 API 只调用，不继承**：业务 hook import 能力 hook 的函数，不 extends
3. **一个交互一个入口**：每个用户操作或系统事件对应业务层的一个明确处理单元
4. **副作用隔离**：setTimeout、Set（branchPendingSessionIds 等 module-level 状态）只在业务层存在

### BL-1：PTY_BIND / PTY_UNBIND 处理

**触发**：`IPC.PTY_BIND` 或 `IPC.PTY_UNBIND` 从主进程推送

#### PTY_BIND 时序

```
收到 PTY_BIND { ptyId, claudeId, transcriptPath, cwd }
  │
  ├─→ [能力 I1] bindPty(ptyId, claudeId)           // 更新双向绑定表
  │
  ├─→ 三路判断（Session 注册表迁移）：
  │     [能力 H5] getSession(ptyId) = byPty?
  │     [能力 H5] getSession(claudeId) = byClaude?
  │
  │   路径 A：byPty 存在（首次启动 bootstrap gap 迁移）
  │     [能力 H1→H2] migrateSessionKey(ptyId, claudeId, {id:ptyId, transcriptPath, cwd})
  │     [能力 J1] 若 ptySessionIds 有 ptyId → removeFromRealtime(ptyId) + addToRealtime(claudeId)
  │
  │   路径 B：byClaude 存在（resume 场景）
  │     [能力 H3] patchSession(claudeId, {id:ptyId, transcriptPath, cwd})
  │     [能力 J1] 若 ptySessionIds 无 claudeId → addToRealtime(claudeId)
  │
  │   路径 C：两者都不存在（外部启动）
  │     [能力 H1] createSession 以 claudeId 为 key，{id:ptyId, claudeId, status:'Running', ...}
  │     [能力 J1] addToRealtime(claudeId)
  │
  └─→ log: `PTY_BIND: PTY ${ptyId} ↔ Claude ${claudeId}`
```

#### PTY_UNBIND 时序

```
收到 PTY_UNBIND { ptyId, claudeId }
  │
  └─→ [能力 I2] unbindPty(ptyId, claudeId)
      // 只清绑定表，不改 session 状态
      // 原因：/branch 时父 PTY 也会 UNBIND，但父 session 还活着（见调试机制文档第十条）
```

### BL-2：Session 生命周期（Hook 事件）

**触发**：`IPC.HOOK_EVENT` 中 eventName = SessionStart / SessionEnd / Stop

#### SessionStart 时序

```
收到 SessionStart { sessionId=claudeId, cwd, transcriptPath, receivedAt }
  │
  ├─→ [能力 H5] getSession(claudeId) = existing?
  │
  │   情况 A：claudeId entry 已存在（PTY_BIND 先到）
  │     [能力 H3] patchSession(claudeId, {transcriptPath, status:'Running'})
  │     返回
  │
  │   情况 B：claudeId entry 不存在，找到 ptyId-keyed 临时 entry（SessionStart 先到）
  │     [能力 H6] findSessionByCwd(cwd) 在 ptySessionIds 范围内查
  │     若找到 matchedKey（ptyId）：
  │       [能力 H3] patchSession(matchedKey, {transcriptPath, status:'Running'})
  │       // key 仍是 ptyId，等 PTY_BIND 再迁移（不在这里改 key）
  │     返回
  │
  │   情况 C：外部启动（无任何 entry）
  │     [能力 H1] createSession(claudeId, {claudeId, status:'Running', transcriptPath, cwd, ...})
  │     返回
```

#### Stop 时序

```
收到 Stop { sessionId=claudeId, cwd }
  │
  ├─→ [能力 H5] getSession(claudeId) = sess?
  │     若 sess && Running → [能力 H3] patchSession(claudeId, {status:'Paused'})
  │     否则 CWD fallback：[能力 H6] findSessionByCwd(cwd) → patchSession(key, {status:'Paused'})
  │
  └─→ [能力 M9] clearWorkStatus(claudeId)   // 清空工作状态词
```

#### SessionEnd 时序

```
收到 SessionEnd { sessionId=claudeId, cwd, receivedAt }
  │
  ├─→ 检查 branchPendingSessionIds.has(claudeId)?（module-level Set，在 BL-3 中维护）
  │
  │   是（branch 拦截）：
  │     branchPendingSessionIds.delete(claudeId)
  │     log: `SessionEnd intercepted for branch on claudeId ${claudeId}`
  │     返回（不改状态）
  │
  │   否（fallback：CWD 匹配找是否有 pending）：
  │     遍历 branchPendingSessionIds，通过 cwd 匹配找到 key → 拦截并返回
  │
  │   正常结束：
  │     [能力 H4] completeSession(claudeId, receivedAt)
  │     若无 claudeId entry → CWD fallback 找到后 complete
  │     [能力 J2] removeFromRealtime(claudeId)
```

### BL-3：Branch 处理

**触发**：`IPC.SESSION_BRANCH_LINK`（预通知 + 确认）、`IPC.JSONL_BRANCH_SNAPSHOT`

**模块级状态**（只在此模块存在）：
```typescript
const branchPendingSessionIds = new Set<string>()   // 等待 branch 确认的父 claudeId
const pendingBranchSnapshots = new Map<string, string>()  // childId → branchStartUuid（竞态缓存）
```

#### 预通知时序（无 childSessionId）

```
收到 SESSION_BRANCH_LINK { parentSessionId=ptyId, (无 childSessionId) }
  │
  ├─→ [能力 I3] resolveClaudeId(parentSessionId) = parentClaudeId
  │     （parentSessionId 是 ptyId，需转为 claudeId）
  │
  └─→ branchPendingSessionIds.add(parentClaudeId)
      // BL-2 的 SessionEnd 会检查此 Set
```

#### 确认通知时序（有 childSessionId）

```
收到 SESSION_BRANCH_LINK { parentSessionId=ptyId, childSessionId=旧claudeId }
  │
  ├─→ [能力 I3] resolveClaudeId(parentSessionId) = parentClaudeId
  │
  ├─→ 防重复：[能力 L4] getBranchRelation(childSessionId) 已存在 → 返回
  │
  ├─→ [能力 K6] getTimelineLength(parentClaudeId) = triggerNodeIndex
  │
  ├─→ [能力 L7] computeBranchSide(parentClaudeId, triggerNodeIndex) = {side, lineLength, branchIndex}
  │
  ├─→ 合并竞态快照：pendingBranchSnapshots.get(childSessionId) = pendingUuid（可能为空）
  │
  ├─→ [能力 L1] registerBranch(childSessionId, parentClaudeId, {
  │       triggerNodeIndex, side, lineLength, branchIndex,
  │       inheritedNodeCount,  // parentTimeline 中 user_input+assistant 节点数
  │       branchStartUuid: pendingUuid
  │     })
  │
  ├─→ 确保 child session 存在：
  │     [能力 H5] getSession(childSessionId) 不存在？
  │       [能力 H5] getSession(parentClaudeId) = parentSession（或 CWD 查找兜底）
  │       [能力 H1] createSession(childSessionId, {
  │           id: childSessionId, claudeId: childSessionId,
  │           projectId: parentSession.projectId,
  │           status: 'Running', cwd: parentSession.cwd, ...
  │         })
  │
  ├─→ [能力 J2] removeFromRealtime(parentClaudeId)
  │     // parent 转入 branch 宿主角色，不再显示在实时面板
  │
  └─→ log: `Branch linked: child=${childSessionId} parent=${parentClaudeId}`
```

#### JSONL_BRANCH_SNAPSHOT 时序

```
收到 JSONL_BRANCH_SNAPSHOT { sessionId=childId, branchStartUuid }
  │
  ├─→ [能力 L4] getBranchRelation(childId) = existing?
  │     是 → [能力 L2] updateBranchSnapshot(childId, branchStartUuid)
  │     否 → [能力 L3] cachePendingSnapshot(childId, branchStartUuid)
  │           // 等 SESSION_BRANCH_LINK 确认通知到达时合并
```

### BL-4：工具活动处理

**触发**：`IPC.HOOK_EVENT` 中 eventName = PreToolUse / PostToolUse / PostToolUseFailure

#### PreToolUse 时序

```
收到 PreToolUse { sessionId=claudeId, cwd, payload: {tool_name, tool_input, agent_id, tool_use_id} }
  │
  ├─→ isSubagentTool = !!payload.agent_id
  │
  ├─→ 构建 entry：entryId, displayText, category, status:'running'
  │
  ├─→ [能力 M1] toolStart(claudeId, cwd, entry)
  │
  ├─→ 若 toolName === 'Agent' → [能力 M4] showSubagent(claudeId, subagentInfo)
  │
  ├─→ 若 !isSubagentTool（主 session 工具调用）：
  │     [能力 K3] appendInsertion(claudeId, buildLineInsertion(...))
  │       若 toolName === 'Agent'：
  │         [能力 M7] n = incrementAgentCount(claudeId)
  │         insertion.badgeContent = {toolName: `Agent${n}:subagentType`, ...}
  │         insertion.toolUseId = payload.tool_use_id  // 精确配对用
  │         insertion.customWidth = 144 + subagentIdsAtom(claudeId).length * 48
  │
  └─→ 若 isSubagentTool（subagent 工具调用）：
        agentId = payload.agent_id
        [能力 K3] appendInsertion(agentId, buildLineInsertion(...))   // 写 agentId 的插入线
        [能力 M6] registerSubagentId(claudeId, agentId)               // 注册到父 session
```

#### PostToolUse 时序

```
收到 PostToolUse { sessionId=claudeId, cwd, payload: {tool_name, agent_id, tool_response, tool_use_id} }
  │
  ├─→ isSubagentTool = !!payload.agent_id
  │
  ├─→ [能力 M2] toolDone(claudeId, toolName)  →  3s 后自动清除 done 条目
  │
  ├─→ 若 toolName === 'Agent' → [能力 M5] hideSubagent(claudeId)
  │
  ├─→ 若 !isSubagentTool：
  │     [能力 K4] updateInsertionStatus(claudeId, toolName, 'done')
  │
  │     若 toolName === 'Agent'（返回任务结果）：
  │       responseText = extractAgentResponse(payload.tool_response)
  │       matchingDispatch = 在 lineInsertionsBySession(claudeId) 中按 tool_use_id 精确查找
  │       agentLabel = matchingDispatch?.badgeContent.toolName ?? 'Agent'
  │       returnInsertion = buildLineInsertion('Agent', {}, ...) + 覆盖 lineLabel/badgeContent
  │       [能力 K3] appendInsertion(claudeId, returnInsertion)
  │
  └─→ 若 isSubagentTool：
        [能力 K4] updateInsertionStatus(agentId, toolName, 'done')
```

#### PostToolUseFailure 时序

```
收到 PostToolUseFailure { sessionId=claudeId, payload: {tool_name, agent_id} }
  │
  ├─→ [能力 M3] toolFailed(claudeId, toolName)  →  5s 后自动清除
  │
  ├─→ 若 !isSubagentTool → [能力 K4] updateInsertionStatus(claudeId, toolName, 'failed')
  └─→ 若 isSubagentTool  → [能力 K4] updateInsertionStatus(agentId, toolName, 'failed')
```

### BL-6：JSONL 数据接收

**触发**：`IPC.JSONL_RECORD` / `IPC.JSONL_RECORDS` / `IPC.JSONL_SUBAGENT_RECORD`

```
JSONL_RECORD { sessionId=rawPtyId, record }
  │
  ├─→ [能力 K5] claudeId = resolveKey(rawPtyId)
  ├─→ node = jsonlRecordToNode(record)
  ├─→ [能力 K1] appendTimelineNode(claudeId, node)
  │
  └─→ 若 record.type==='assistant' && 包含 Insight 块：

        [能力 M8] setInsight(record.sessionId, insightText)  // AgentBlock 金色卡片


JSONL_RECORDS { sessionId=rawPtyId, records[] }
  │
  ├─→ [能力 K5] claudeId = resolveKey(rawPtyId)
  ├─→ nodes = records.map(jsonlRecordToNode).filter(Boolean)
  ├─→ [能力 K2] appendTimelineNodes(claudeId, nodes)
  │

JSONL_SUBAGENT_RECORD { sessionId=rawPtyId, agentId, record }
  │
  ├─→ [能力 K5] claudeId = resolveKey(rawPtyId)
  ├─→ node = jsonlRecordToNode(record)
  ├─→ [能力 K1] appendTimelineNode(`${claudeId}::${agentId}`, node)
  └─→ [能力 M6] registerSubagentId(claudeId, agentId)
```

## IpcBridge 入口组装（App.tsx 挂载）

重构后，`useIpcBridge.ts` 变为一个**组合入口**，只负责把各业务处理单元的订阅注册到 IPC 监听：

```typescript
// useIpcBridge.ts（重构后，~60 行）
export function useIpcBridge(): void {
  const store = useStore()

  // 初始化各业务处理单元（每个返回 unsubscriber 数组）
  const handlers = [
    usePtyBindHandler(store),       // BL-1
    useSessionLifecycle(store),     // BL-2（内部持有 branchPendingSessionIds）
    useBranchHandler(store),        // BL-3（内部持有 pendingBranchSnapshots）
    useToolActivityHandler(store),  // BL-4
    useSubagentHandler(store),      // BL-5
    useJsonlHandler(store),         // BL-6
    useStatusLineHandler(store),    // BL-7
    usePermissionHandler(store),    // BL-11
    useContextHandler(store),       // BL-12
  ]

  useEffect(() => {
    // 加载项目列表
    void window.api.invoke(IPC.PROJECT_LIST).then((raw) => {
      store.set(projectsAtom, new Map((raw as Project[]).map((p) => [p.id, p])))
    })

    // 注册所有 IPC 监听
    const unsubscribers = handlers.flatMap((h) => h.register())
    return () => unsubscribers.forEach((u) => u())
  }, [store])
}
```

## 关键跨模块约束（业务层必须遵守）

### 约束 1：BL-2 和 BL-3 共享 branchPendingSessionIds

`branchPendingSessionIds` 是 module-level Set，必须在 **同一个闭包作用域**中存在。

实现方式：将 BL-2 和 BL-3 合并到同一个文件，或将 Set 通过参数传递：

```typescript
// 方案：组合入口时传递共享状态
const branchPendingIds = new Set<string>()
useSessionLifecycle(store, branchPendingIds)   // BL-2 读此 Set
useBranchHandler(store, branchPendingIds)       // BL-3 写此 Set
```

### 约束 2：BL-1 的三路判断顺序不能改变

byPty 判断必须在 byClaude 之前（见调试机制文档第二条）。

### 约束 3：BL-3 确认通知中 registerBranch 必须在 removeFromRealtime 之前

确保 `sessionRelationsAtom` 写入完成后再从实时面板移除父框，避免 LeftPanel 的 `isBranchSession` 判断在移除时读到旧 relation。

### 约束 4：SESSION_STATUS 的 sessionId 是 ptyId，需转换

`IPC.SESSION_STATUS` 处理逻辑（在 BL-2 或单独的 useSessionStatusHandler 中）：
```typescript
const claudeId = resolveClaudeId(rawId)  // 能力 I3
const key = claudeId ?? rawId            // fallback：bootstrap gap 期间
```

### 约束 5：JSONL 事件的 sessionId 也是 ptyId

BL-6 中每次写 timeline atom 之前必须调用 K5 `resolveKey(rawId)`。
