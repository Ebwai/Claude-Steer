# 调试过程中发现的机制

> 本文记录 `useIpcBridge.ts` 及相关模块在实际调试过程中发现的非直觉机制、竞态规律和必须遵守的不变量。
> 重构时这些规律必须全部保留，否则会重现已修复的 bug。

---

## 一、双 ID 系统：ptyId vs claudeId

### 本质区别

| 字段 | 含义 | 生命周期 | 来源 |
|---|---|---|---|
| `ptyId` | node-pty 进程 UUID | 每次 `SESSION_START` / `SESSION_RESUME` 都是新值 | 主进程 `PtyManager.startSession()` 生成 |
| `claudeId` | Claude Code 内部 session UUID | 跨 resume **永远不变** | `SessionStart Hook` 的 `hookEvent.sessionId` |

### 关键推论

- `claude -r <claudeId>` 恢复后，claudeId **不变**，但 ptyId **是一个新的值**。
- `activeSessionsAtom` 的 Map key **必须用 claudeId**，否则每次 resume 都会创建新节点（React 认为是新元素，触发完整重建）。
- `ptyId` 只用于实际 PTY 操作（`SESSION_STOP`、`TERM_WINDOW_OPEN` 的 IPC payload）。

---

## 二、PTY_BIND 三路判断（核心逻辑，禁止简化）

PTY_BIND 到达时，`activeSessionsAtom` 中可能存在三种状态之一：

```
byPty    = next.get(ptyId)     // 临时 entry（LeftPanel/ProjectSettingsBar 立即创建，key=ptyId）
byClaude = next.get(claudeId)  // 已有 claudeId entry（resume 场景：entry 已是 claudeId key）
```

### 路径 1：byPty 存在（首次启动的 bootstrap gap 迁移）

- **触发场景**：LeftPanel 在 `SESSION_START` 返回后立即以 ptyId 为 key 创建了临时 entry（为了不让 UI 有空白延迟）
- **操作**：`next.delete(ptyId)` + `next.set(claudeId, {...byPty, id: ptyId, claudeId})`
- **结果**：key 从 ptyId 迁移到 claudeId，node.id 变为 claudeId

### 路径 2：byClaude 存在（resume 场景）

- **触发场景**：`claude -r` 之后，旧 session entry 以 claudeId 为 key 已存在（status=Paused）
- **操作**：`next.set(claudeId, {...byClaude, id: newPtyId, claudeId})`
- **结果**：claudeId key 不变，`id` 字段更新为新 ptyId，node.id = claudeId → **零 React 重建**

### 路径 3：两者都不存在（外部启动）

- **触发场景**：用户在终端直接运行 `claude`，未经过 LeftPanel
- **操作**：直接 `next.set(claudeId, {id: ptyId, claudeId, ...})`

### 关键约束

- 三路判断顺序不能颠倒：byPty 优先于 byClaude，否则 resume 时如果恰好 ptyId 碰撞会走错路径（实践中极少，但逻辑上必须正确）。
- PTY_BIND **不能**读取 `sessionRelationsAtom` 来做判断——因为 PTY_BIND 必然早于 SESSION_BRANCH_LINK 到达（见下文），关系还没写入，读到的永远是空。

## 三、ptySessionIdsAtom 的迁移时机

`ptySessionIdsAtom` 存的是 **claudeId**（迁移完成后），用于 LeftPanel 过滤"真实 PTY session"。

迁移发生在 **PTY_BIND** 时：

```typescript
if (next.has(ptyId)) {
  next.delete(ptyId)
  next.add(claudeId)  // ptyId → claudeId
  return next
}
if (!next.has(claudeId)) {
  next.add(claudeId)  // 外部启动：直接加 claudeId
}
```

- LeftPanel 在 `SESSION_START` 后立即 `add(ptyId)` → PTY_BIND 时迁移为 claudeId。
- `ptySessionIdsAtom` 同时承担两个语义：**"有 PTY 绑定"** 和 **"在实时面板显示"**。为了解耦这两个语义，/branch 确认时主动从中删除 parentClaudeId（见下文）。

---

## 四、/branch 触发链路（两阶段协议）

### 事件到达顺序（固定，基于主进程同步链路）

```
1. PTY_BIND { ptyId=父PTY, claudeId=新ClaudeId }         ← 必然先到
2. SESSION_BRANCH_LINK 预通知 { parentSessionId=父PTY }   ← 后到
3. SessionEnd Hook { sessionId=旧ClaudeId }               ← 父 session 的 REPL 结束
4. SESSION_BRANCH_LINK 确认通知 { parent=父PTY, child=旧ClaudeId }
5. SessionStart Hook { sessionId=子ClaudeId }             ← 子 session 开始
```

**根本原因**：主进程 `autoWatchTranscript → tryRegister → bindPtyToClaudeSession → PTY_BIND 推送` 是同步链路，在它返回后才发 SESSION_BRANCH_LINK。

### 预通知（无 childSessionId）的唯一作用

设置 `branchPendingSessionIds.add(parentClaudeId)`，目的是**拦截即将到来的 SessionEnd**，防止 parent session 被标记为 Completed（PTY 还活着）。

```typescript
if (!childSessionId) {
  branchPendingSessionIds.add(parentClaudeId)
  return  // 仅此，不做其他操作
}
```

### SessionEnd 拦截逻辑

```typescript
if (branchPendingSessionIds.has(hookEvent.sessionId)) {
  branchPendingSessionIds.delete(hookEvent.sessionId)
  intercepted = true
  // 跳过 Completed 状态变更
}
```

- `branchPendingSessionIds` 存 claudeId（与 SessionEnd 的 `hookEvent.sessionId` 一致）。
- 有一个 CWD fallback：极少数竞态下 sessionId 匹配失败，通过 cwd 匹配找到 pending entry。

### 确认通知的副作用（顺序重要）

1. 写入 `sessionRelationsAtom`（key=childClaudeId, parentSessionId=parentClaudeId）
2. 确保 child session 在 `activeSessionsAtom` 中（SessionStart Hook 可能尚未到达）
3. 从 `ptySessionIdsAtom` 删除 `parentClaudeId`（父框从实时面板消失，转为 branch 宿主角色）

## 五、SESSION_BRANCH_LINK 的 parentSessionId 是 ptyId，不是 claudeId

IPC 消息里的 `parentSessionId`（或 `sessionId`）是父 session 的**当前 ptyId**，不是 claudeId。

必须通过绑定表转换：

```typescript
const parentClaudeId = store.get(ptyBindingsAtom).ptyToClaudeMap.get(parentSessionId) ?? parentSessionId
```

- `?? parentSessionId` 是 fallback：极少数情况下 PTY_BIND 还未到达（理论上不应发生，但防御性保留）。
- `sessionRelationsAtom` 的 value 里 `parentSessionId` 存的是 **parentClaudeId**（已转换后的值），确保 canvas edge source/target 均为 claudeId = node.id。

---

## 六、sessionRelationsAtom 的 key 是 childClaudeId，禁止迁移

- Map key = childSessionId（旧 branch claudeId）
- value.parentSessionId = parentClaudeId

禁止在 PTY_BIND 时把 relation 的 key 从 childClaudeId 迁移到 childPtyId——这会造成自引用，破坏 canvas 连线。

LeftPanel 查询"某 session 是否是 branch 宿主"用**反向遍历**：

```typescript
const isBranchSession = Array.from(sessionRelations.values()).some(
  (r) => r.type === 'branch' && r.parentSessionId === (sess.claudeId ?? sess.id)
)
```

不用 `sessionRelations.get(sess.id)`——用 ptyId 当 childId 去查永远找不到。

---

## 七、resolveTimelineKey：JSONL 事件的 ptyId → claudeId 转换

JSONL 事件（`JSONL_RECORD`、`JSONL_RECORDS`、`JSONL_SUBAGENT_RECORD`）的 `sessionId` 字段来自主进程 `autoWatchTranscript`，是 **ptyId**。

但 `timelineBySessionAtom` / `lineInsertionsBySessionAtom` 的 key 与 `ProcessTimeline` 读取时使用的 `SessionFrameNode.data.sessionId`（= claudeId）必须一致。

因此每次写 timeline atom 之前都要转换：

```typescript
function resolveTimelineKey(ptyId: string): string {
  return store.get(ptyBindingsAtom).ptyToClaudeMap.get(ptyId) ?? ptyId
}
const sessionId = resolveTimelineKey(rawId)
```

- `?? ptyId` 是 fallback：PTY_BIND 尚未到达时（极少），先用 ptyId 写入，等 PTY_BIND 后绑定表建立，后续写入就会用 claudeId key，与前面的 ptyId key 的数据共存（已知已接受的小 bug，实际影响极小）。

---

## 八、claude -r 不触发 SessionStart Hook

`claude -r <claudeId>` 恢复 session 时：
- **不触发 SessionStart Hook**（resume 专有路径）
- 唯一的绑定时机是主进程 `autoWatchTranscript → tryRegister`（检测到新 JSONL 文件）
- 此时发送 `PTY_BIND { ptyId=新PTY, claudeId=原claudeId }`
- 渲染层通过 PTY_BIND 路径 2（byClaude）更新 session entry 的 `id` 字段为新 ptyId

---

## 九、SESSION_STATUS 的 sessionId 是 ptyId

`IPC.SESSION_STATUS` 事件（PTY 层推送）的 `sessionId` 是 **ptyId**，不是 claudeId。

必须通过绑定表转换后才能更新 `activeSessionsAtom`：

```typescript
const claudeId = store.get(ptyBindingsAtom).ptyToClaudeMap.get(rawId)
const sessionKey = claudeId ?? rawId  // fallback：PTY_BIND 前用 ptyId 查（bootstrap gap）
```

## 十、PTY_UNBIND 只清绑定表，禁止改 session 状态

`PTY_UNBIND` 到达时：
- **只**删除 `ptyBindingsAtom` 里的双向映射
- **不**把 session 标记为 Completed 或 Interrupted

原因：/branch 时，父 session 的旧 PTY 也会走 PTY_UNBIND，但父 session 还活着（PTY 换了个新进程继续运行）。如果 UNBIND 时改状态，会导致 AgentBlock 消失。

session 状态变更只能由 **SessionEnd Hook** 触发（且受 branchPendingSessionIds 拦截保护）。

## 十一、JSONL_BRANCH_SNAPSHOT 竞态

`JSONL_BRANCH_SNAPSHOT { sessionId: childId, branchStartUuid }` 可能**早于** `SESSION_BRANCH_LINK` 到达（文件扫描比 IPC 事件快的极端情况）。

处理方式：缓存在 `pendingBranchSnapshots = new Map<string, string>()`，等 SESSION_BRANCH_LINK 确认通知到达时合并写入 `relation.branchStartUuid`。

```typescript
const pendingUuid = pendingBranchSnapshots.get(childSessionId)
if (pendingUuid) pendingBranchSnapshots.delete(childSessionId)
// 写入 relation 时携带 branchStartUuid: pendingUuid
```

---

## 十二、subagentIdsAtom Hook 优先注册

subagent 的 agentId 需要注册到父 session 的 `subagentIdsAtom`，有两个来源：

1. **PreToolUse Hook**（`payload.agent_id` 存在时）
2. **JSONL_SUBAGENT_RECORD**（首次收到 subagent 记录时）

**必须让 Hook 优先**：JSONL 文件写入比 Hook 触发慢（Hook 是同步 HTTP 回调，JSONL 是异步文件写入），依赖 JSONL 会有竞态。PreToolUse 一定先于 JSONL 写入触发，用它注册 agentId 是最强约束。

---

## 十三、Agent 工具调用的 tool_use_id 精确配对

并行 Subagent 的"分配任务"和"返回任务结果"插入线必须用 `tool_use_id` 精确匹配，禁止用顺序（FIFO）匹配。

原因：两个 Agent 同时运行时，返回顺序不确定（可能是 Agent2 先返回），FIFO 会把返回结果错误地归给 Agent1。

```typescript
// PreToolUse 时存 toolUseId
insertion.toolUseId = payload.tool_use_id

// PostToolUse 时精确查找
const matchingDispatch = store.get(lineInsertionsBySessionAtom(ptyId))
  .find((i) => i.type === 'subagent' && i.lineLabel === '分配任务' && i.toolUseId === payload.tool_use_id)
```

## 十四、statusLine 兜底：transcriptPath 补全

触发条件：`SessionStart Hook` 未触发（stream-json 恢复模式等极端情况），session 没有 transcriptPath。

兜底逻辑：statusLine 每 ~300ms 推送一次，携带 `transcript_path` 和 `cwd`。通过 `session.cwd === statusLine.cwd` 精确匹配，为缺少 transcriptPath 的 Running session 补全。

**安全约束**：必须是精确 cwd 匹配，不用 startsWith，防止跨项目污染。

---

## 十五、Stop Hook 的 CWD fallback（bootstrap gap 期间）

Stop Hook 到达时，`hookEvent.sessionId` 是 claudeId，但在 bootstrap gap 期间（LeftPanel 创建了 ptyId key 的临时 entry，PTY_BIND 尚未到达），`activeSessionsAtom` 里还没有 claudeId key。

处理：先尝试直接 `next.get(hookEvent.sessionId)` 查 claudeId，失败则 CWD 匹配兜底。

```typescript
const sess = next.get(hookEvent.sessionId)  // claudeId 直接查
if (sess && sess.status === 'Running') {
  // 正常路径
} else {
  // CWD fallback：bootstrap gap 期间 key 仍是 ptyId
  for (const [key, s] of next) {
    if (s.status === 'Running' && s.cwd 匹配) { ... break }
  }
}
```

SessionEnd 的 CWD fallback 逻辑完全相同。

## 十六、/branch 的 30s 冷却防重（主进程侧）

PTY stdout 的 "Branched─conversation" 字符串在 branch 确认后仍会多次出现（回显等），主进程用 `confirmedBranchPtyIds = new Set<string>()` 防重：

- branch 确认后 `confirmedBranchPtyIds.add(parentPtyId)`
- `setTimeout(() => confirmedBranchPtyIds.delete(parentPtyId), 30_000)`
- 检测 stdout 时先 `!confirmedBranchPtyIds.has(sid)` 判断

不加此防重会在 LeftPanel 显示多个重复的 AgentBlock。

## 十七、ptySessionIdsAtom 的双重语义（已知问题，重构时需拆分）

当前 `ptySessionIdsAtom` 同时承担：

1. **"有 PTY 绑定的 session"**（能力语义）
2. **"应显示在实时面板的 session"**（业务语义）

这两个语义耦合导致了 /branch 后父框永久显示 Paused 的 bug（已用手动 delete 打补丁）。

**根本解法**（重构时实现）：把"是否在实时面板显示"变成 derived atom，由 `activeSessionsAtom + sessionRelationsAtom` 计算派生，而不是靠手动维护集合。

---

## 十八、SessionStart Hook 的三种到达时序

SessionStart Hook 与 PTY_BIND 的相对顺序不固定：

- **情况 A**：PTY_BIND 先到（多数情况）→ `activeSessionsAtom` 中已有 claudeId entry → SessionStart 只更新 transcriptPath
- **情况 B**：SessionStart 先到（约 5% 概率）→ `activeSessionsAtom` 中只有 ptyId 的临时 entry → SessionStart 通过 CWD 匹配找到并更新 transcriptPath（key 保持 ptyId，等 PTY_BIND 再迁移）
- **情况 C**：外部启动（无 LeftPanel 临时 entry）→ SessionStart 直接创建 claudeId entry

因此 SessionStart 处理中必须同时处理三种情况，不能假设一定有或没有 claudeId entry。
