# 能力层设计文档

> **定位**：能力层提供原子化的、无业务语义的功能单元。
> 每个能力组只管理自己领域的状态，提供稳定 API，不主动调用其他能力组，不含时序编排逻辑。
> 业务层通过调用这些能力 + 设计时序完成具体交互。

---

## 设计原则

1. **原子性**：每个能力单元只做一件事，可独立测试
2. **无跨域依赖**：能力模块不 import 其他能力模块（主进程端允许 PtyManager 等互相依赖，但渲染端 hooks 严格隔离）
3. **稳定契约**：API 不随业务逻辑变化而变化，业务层改了只改调用方
4. **状态自治**：每个能力组只读写自己负责的 atom/状态，不越界

## 主进程端能力组

### A. PTY 进程能力组 — `PtyManager.ts`

| 能力 | API | 入参 | 返回 | 说明 |
|---|---|---|---|---|
| A1 启动 PTY | `startSession(opts)` | `{projectId, projectPath, permissionMode, model?}` | `{sessionId: ptyId}` | 生成 UUID, spawn claude CLI, 注册 onData/onExit |
| A2 停止 PTY | `stopSession(ptyId)` | ptyId | void | 2×Ctrl+C → 500ms → SIGTERM → SIGKILL |
| A3 恢复 PTY | `resumeSession(opts)` | `{projectPath, resumeSessionId: claudeId, permissionMode}` | `{sessionId: newPtyId}` | spawn `claude -r <claudeId>` |
| A4 写入 stdin | `writeToSession(ptyId, text)` | ptyId, string | void | 写入消息队列，防并发 |
| A5 调整终端尺寸 | `resizeSession(ptyId, cols, rows)` | ptyId, number, number | void | 同步 xterm.js 和 PTY 终端尺寸 |
| A6 获取 PTY 列表 | `getActiveSessions()` | — | Map<ptyId, PTYInfo> | 用于反查 CWD 等 |

**状态约束**：
- A2 停止后不主动推送任何 IPC，session 状态变更由 SessionEnd Hook 处理
- A3 恢复后不触发 SessionStart Hook（已验证），绑定依赖 autoWatchTranscript
- 权限模式 → CLI 参数映射：`acceptEdits`→`--allowedTools Edit,Write,MultiEdit`；`plan`→`--plan`；`auto/bypassPermissions`→`--dangerously-skip-permissions`

### B. Session ID 绑定能力组 — `index.ts bindPtyToClaudeSession`

> 双向映射表：`ptyToClaudeMap: Map<ptyId, claudeId>` + `claudeToPtyMap: Map<claudeId, ptyId>`

| 能力 | 函数 | 说明 |
|---|---|---|
| B1 建立绑定 | `bindPtyToClaudeSession(ptyId, claudeId, transcriptPath, cwd)` | 写双向表，推送 `PTY_BIND` 到渲染层 |
| B2 解除绑定 | `unbindPtyFromClaudeSession(ptyId, claudeId)` | 清双向表，推送 `PTY_UNBIND` |
| B3 pty→claude 查询 | `ptyToClaudeMap.get(ptyId)` | 渲染层通过 `ptyBindingsAtom` 使用 |
| B4 claude→pty 查询 | `claudeToPtyMap.get(claudeId)` | 同上 |
| B5 早绑定（autoWatchTranscript） | `tryRegister(newFile, ptyId, isPtyId, parentPtyId?)` | 检测到 JSONL 新文件 → 触发 B1 |

**约束**：
- B2 只清绑定表，**禁止**改 session 状态（/branch 时旧 PTY 也走 UNBIND，但父 session 还活着）
- B5 是 resume 后的唯一绑定时机（`claude -r` 不触发 SessionStart Hook）

### C. Hook 事件接收能力组 — `HookServer.ts` + `HookEventBus.ts`

| 能力 | 说明 |
|---|---|
| C1 启动 HTTP 服务器 | 监听 39521 端口，/hooks + /statusline 两个端点 |
| C2 解析分发 Hook 事件 | payload → HookEvent → `IPC.HOOK_EVENT` push 到渲染层 |
| C3 接收 StatusLine 推送 | ~300ms 一次，推送 `IPC.STATUS_LINE` |
| C4 端口冲突通知 | 39521 被占用时弹窗提示用户 |

**支持的 Hook 事件类型**：`SessionStart / SessionEnd / Stop / PreToolUse / PostToolUse / PostToolUseFailure / SubagentStart / SubagentStop / PermissionRequest / PermissionDenied / Notification / PreCompact / PostCompact`

### D. JSONL 监听能力组 — `JsonlWatcher.ts` + `JsonlParser.ts`

| 能力 | API | 说明 |
|---|---|---|
| D1 注册实时监听 | `watchFile(path, sessionId, onRecord)` | tail 模式：只读新增内容，offset 递增 |
| D2 全量历史读取 | `readFromStart(path, sessionId, onBatch)` | 每批 50 条，推送 `JSONL_RECORDS` |
| D3 Branch 快照检测 | 内置于 D2/D1 | 扫描最后 30 行找 `file-history-snapshot` → 推送 `JSONL_BRANCH_SNAPSHOT` |
| D4 Subagent 文件发现 | `scanSubagentsDir(dir)` | 深度 3，识别 `subagents/agent-*.jsonl` |
| D5 单行 JSONL 解析 | `parseJsonlLine(line)` → `JsonlRecord` | 含 type/text/toolUse/toolResult/sessionId/agentId/isBranchStart |

**约束**：
- D1 使用 inotify（chokidar）监听，`ignoreInitial: true`（否则已有文件也触发 add 事件）
- D4 必须 depth:3 才能覆盖 `subagents/<uuid>/` 层级
- D5 中 `isBranchStart` 由 `file-history-snapshot` 消息标记，表示该条是 branch 分叉起点

## 渲染进程端能力组

> 渲染端能力组以 Jotai atom + 操作函数的形式存在，每组对应一个专用 hook。
> 重构后每个 hook 只持有一类 atom，通过 `store.get/set` 操作。

### H. Session 注册表能力组 — `useSessionRegistry.ts`（待创建）

**管理的 atom**：`activeSessionsAtom: Map<claudeId, Session>`

| 能力 | 操作 | 说明 |
|---|---|---|
| H1 创建 Session entry | `createSession(ptyId, opts)` | 以 ptyId 为临时 key 创建（bootstrap gap 阶段） |

| H6 查询 by CWD | `findSessionByCwd(cwd)` → `[claudeId, Session]` \| undefined | 遍历 fallback |
| H7 查询 by ptyId | `findSessionByPtyId(ptyId, bindings)` → Session \| undefined | 通过绑定表转换 |
