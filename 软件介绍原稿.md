            -  CLI

        - subagent工作框：主要关注的是subagent的一些行为，以及其与主agent之间的交接（subagent要干的任务&subagent返回给主agent的结果）

  无论您是想要一个可视化CC的命令行界面，还是想学习agent的机制或AI工作本身，或是想要一个多agent多项目的管理方法，又或者您想尝试harness AI并记录得到反馈等等下列需求，那么都建议您尝试一下本软件。这么说可能有些抽象，下面会结合案例深挖一下这些需求和痛点

虽然CLI的形式受到很多技术大佬的喜欢，但是主包目前还是认为图形化相比于CLI才是更适合人类的人机交互方式，图形化的界面，有价值的信息密度更高，重点更突出，也方便各种点按交互【但似乎CLI和GUI的争论点就在于点按会不会影响心流这个点】，比如这里是Claude Code原生的终端界面，在没有Ctrl+O展开的时候，，密密麻麻的文字就已经让人心烦了，展开之后更是容易迷失在大量信息中抓不住重点。【CLI非图形化界面】。假如出现subagent和多agent的信息，在原生终端观察起来会更加的不直观
或者我们也想了解agent的原理，比如基于react的工作流程，了解上下文加载，何时会调用tool，调用什么tool，如何派发多个subagent工作的等等
又或者我们想学习汇聚了全人类智慧的 AI 是如何解决问题的，如何思考，如何行动的，
这些都是受制于原生claude code的命令行文字界面，没有可视化，所不容易去满足的需求和痛点

在我们尝试了几次vibe coding之后，那么自然会接触到多项目与多 Agent 管理来提高效率，充分压榨token plan的额度，但因为claude code原生的是一个会话对应一个窗口，并行开发多个项目，每个项目有多个agent的时候，会发现不知道哪个命令行窗口对应哪个，主包之前没用claude steer并行开发的时候就经常浪费很多时间在找对话上，在多个命令行窗口中频繁切换，再加上每个agent有不同的工作过程和工作要求，不止会打断心流，甚至我们会出现不同窗口之间弄混淆的情况，比如把任务B的内容发送给了负责任务A的进程的命令和窗口中。更不用说有时候因为一些事情有一段时间没有vibe coding了，想继续回到之前的项目，却发现找不到上次工作的命令行窗口历史了，得从头输入上下文再来一遍（浪费token，浪费时间）【多项目统筹管理，多session混乱治理，多 Agent 协同作业】

【单项目深度管理】

同时，在每个项目中，我们也需要一定的深度管理，
首先因为每个项目或者每个agent所负责的任务不同，我们对每个项目，每个agent有不同的skills和配置permission要求是不一样的，需要精细化的配置，但是原生的这些配置，要么在命令行中的/config里配置，要么在项目的.claude文件夹下去配置，因为步骤有些复杂，我们甚至明明有好用的skills，但是也会因为手头工作太忙，懒得去配置，甚至不知道应该怎么配置

在项目的具体执行中，claude code其实也是自带了一些概念可以帮助我们更好的与Agent交互的，
比如btw的作用就是直接复用对话的上下文cache，单次回答你上下文中有关的问题，不会进行action，且不会增加原有对话的上下文。因为复用缓存所以token消耗小，同时也不会出现agent调用了一大堆工具来回答你简单问题的情况，
branch也是cc的一个概念，就是从当前会话的agent分出一个新的独立agent，继承原始agent的所有上下文，这样就可以有2个具有相同上下文的agent可以同时并行工作了
还有一个在cc中常用的技巧是subgent概念，subagent的意义主要为了不污染主线agent的上下文，一些会产生大量低价值信息的场景比如有探索类场景，探索类中主线agent只关心一个结果，而探索过程中产生的信息对主线agent来说是意义不大的，这些苦力活就适合派发给subagent

在单个项目的具体开发中，由于我们不知道，或者不知道何时用，怎么用，往往不会用上这些技巧，但这些技巧往往对于人来Harness AI非常重要，我们需要在需要用到这些技巧的时候知道这些技巧并一键使用。
并且claude code对于这些概念的可视化我个人觉得是有些捡漏的，比如btw的问答无法长期保存（明明我们临时的提问说明了是我们未知的有价值点），branch时，无法在一个界面同时观察和管理2个agent的行动，subagent也很难看到具体在做什么，又返回了什么信息给claude code
【通知统一】CC原生的Agent执行过程中的权限申请也是一个打乱我们心流的点，这里有2个情况会打断心流，一个是用户在干自己的事情，不想一直盯着agent的工作去浪费注意力在agent有没有被权限卡住而没有在工作；还有一个是多个项目，每个项目多agent在进行的时候，总会有些权限相邻很近的时间出现，一个一个的切换去确认是不是在阻塞，既打断了自己的心流，也耽误了agent因权限同意而阻塞了原来的工作



【计划、可靠性与效率】
关于计划、可靠性与效率，
【plan】【局部的视野带来的偏差】众所周知，我们在日常使用AI的时候，AI第一次给我们的输出大概率是不会让我们完全满意的，即使我们在过程中会根据中间结果不断的介入调整，在复杂的长任务中，最终的输出也难免和我们最初的设想存在一定偏差的，这个通常是因为我们在agent过程中只具有局部的视野，而并没有一个全局的视野。
【分割大型任务】并且Agent的过程可以类似于一个搭积木的过程，假如agent前面有一步没有做好，慢慢累积，直到搭建到某一块积木的时候，发现怎么也无法继续往上搭建了，甚至需要全部重新开始才能继续工作
所以在vibe coding时，为一个需要长期工作的项目都工作过程制定一个计划是非常重要的，不仅可以让我们具有上帝视角作为管理者统筹规划agent的工作进度，也可以避免我们工作在半途中发现错误太多只能重开的情况。
还是回到我们刚刚积木的例子，不是有了plan就可以了，而是需要多级plan，多级plan的每一级就像是一块积木，而没有plan或plan颗粒度过于大，就会像视频中的这样，搭建到了一定的高度出现了问题，但是因为没有一个一个plan的阶段性成果进行区分，即使返工也不知道应该回到哪里进行返工

【git，版本回滚】有了多级 plan 进行目标路径分割，也需要 git 版本控制作为工具，回到偏移的特定点位版本回滚，然后进行微调。阶段性的大成果最好还可以和云端同步进度

【上下文】为什么AI会产生偏差呢？AI 偏差的原因之一了来自大模型的幻觉，比如你问一个AI，今年世界杯梅西进了几球，那么有可能发生的事情，是AI查了2022年世界杯的新闻，或者甚至查都没查，就直接给了你回答，那这个回答显而易见大概率是错误的，这就是AI幻觉的导致的，大模型在概率上觉得这么回答是对的。所以时刻关注对应agent的上下文是一件非常重要的事，如果AI看都没看你给的资料就给了回复，那么这时候就要注意AI接下来的回复是不是编了一个看似非常正确的答案来忽悠我们了

【成本与配置】
关于成本与配置，我们大部分人在vibe coding中都会订阅token plan或者用各个厂家提供的api，这时我们就想知道每月花了多少token，每天花了多少token，每个项目花了多少token，每个对话花了多少token，哪个模型我用的最多。

并且因为我们可以订阅了很多家的token plan或单纯就是想换不同的模型去体验，那么快速切换不同的api provider也会是一个刚需

Claude Code自带的配置项散落的到处都是，有的在用户级别的.claude文件夹，有些在项目级别的.claude文件夹，有些在setting.json里，有的在命令行中的/config中配置，即使找到了配置的地方，我们也不知道应该配置哪个，实际上大部分的设置我们都是不需要去修改的，知道哪些设置会经常修改，以及有一个入口可以快速修改也是一个需求

【复盘】

-在碎片化使用 Claude Code 时，很难手动整理交互过程形成系统性总结，或者在大多数使用agent的时候本身手头上就有其他任务正在做，根本没有时间去复盘用AI的经验或学习AI本身的思考和工作过程。我们需要一个东西帮我们自动复盘，总结经验，提升工作效率
我们也希望我们与claude code的所有对话都可以有条有理的整理好，有书签，有文件夹，方便我们对历史交互进行回顾和复盘，或返回到之前的工作中继续工作。以及万一 AI 犯了错，可以追溯到具体哪一步开始出错、怎么出的错

【远程】
有时候我们在地铁上，上厕所的时候会迸发出一些灵感，但是恰好电脑不在身边，时间长了，灵感或许就忘记了。又或者我们原本在干的任务因为一些其他事情打断了，要离开电脑一段时间，但是我们依然想知道agent的进度。这些情况下我们都需要一个可以远程连接自己常用电脑上claude code的方式，可以继续发号指令给agent，也可以知道agent现在的工作进度。并且等回到电脑旁边的时候还可以继续回到远程时没做完的工作中继续。
